## Ключевые находки прогона

1. **Санитизация вместо отказа (сц. 73/74):** income/expense/donation с `participant_id`
   сервер форсирует `participant_balance_kind = 'none'` (178:457) — клиент физически не может
   передать блок. Долг участника не трогается → инвариант ТЗ 10.13 соблюдён. «→ отказ» из
   формулировки сценария закрыт структурно (невозможностью опасного ввода), не хардкод-ошибкой.
2. **Гэпы закрыты (миграция 200):** `fin_update_payer` (сц.68) и ветки доступа
   `fin_v_attachments` для пользователя счёта (сц.89/77).
3. **Три моих фикстурных ошибки** (не баги кода): сц.8 ждал kassa=20000 (верно 12000 —
   8000 реально потрачены); сц.18 расход на непрофинанс. реальный счёт (верно insufficient_funds,
   сц.21); сц.93 provisioning упёрся в FK.

## Осталось до полного зелёного

- **K2** — как только Адриан даст DB-пароль (Dashboard → Settings → Database).
- **сц.93 полный** — в shadow-периоде, когда департаменты получат реальные аккаунты
  (присоединяется к 📋-бакету вместе с 26/36).

Код-гейты K2/93 подтверждены чтением (FOR UPDATE / ORDER BY id / NOT v_is_admin AND
fin_account_access по всем RPC); не хватает только живого параллелизма/реального юзера.
