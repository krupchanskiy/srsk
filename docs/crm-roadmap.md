# CRM — Детальный план развития

> Цель: сделать CRM таким, чтобы менеджер мог обработать заявку от первого контакта до заселения, не покидая систему и не забыв ни одного шага. А руководитель — за 30 секунд понял, где проблемы.

---

## Типичные рабочие ситуации

Прежде чем проектировать, разберём реальные сценарии:

### Сценарий 1: Новая заявка с сайта
Гость заполнил форму → CRM создала сделку (lead) → менеджер видит на канбане → открывает карточку → звонит/пишет в WhatsApp → записывает результат → меняет статус на «Контакт установлен» → выясняет потребности (проживание, трансфер, виза) → отправляет счёт → отмечает оплату → бронирует комнату.

**Что нужно:** быстрый доступ к контактам, удобный лог звонков, чеклист «что выяснить».

### Сценарий 2: Семья из 5 человек
Муж заполнил форму → менеджер узнаёт, что едет с женой и 3 детьми → записывает состав → выставляет один счёт на всех → одна комната → один трансфер.

**Что нужно:** блок «состав группы» в карточке, без создания 5 отдельных сделок.

### Сценарий 3: Повторный гость
Человек был на прошлом ретрите → подал заявку снова → менеджер должен сразу это увидеть и сказать: «Рады видеть вас снова! В прошлый раз вы жили в комнате X, хотите так же?»

**Что нужно:** блок «история гостя» показывается ДО первого звонка, с деталями прошлых визитов.

### Сценарий 4: Гость передумал
Гость сказал «не поеду» → менеджер отменяет сделку → указывает причину → через месяц система напоминает: «Предложить следующий ретрит?»

**Что нужно:** причина отмены + автозадача на follow-up.

### Сценарий 5: Гость из Индии
Живёт в Дели → виза и билеты не нужны → может не нужен трансфер → только проживание и оплата.

**Что нужно:** чеклист с вариантом «не нужно», чтобы менеджер не забыл отметить (а не просто оставил пустым).

### Сценарий 6: Руководитель проверяет работу
Утром открывает CRM → хочет знать: сколько новых заявок без ответа? Кто из менеджеров загружен? Сколько денег собрали на ретрит? Есть ли просроченные задачи?

**Что нужно:** дашборд с ключевыми метриками, фильтр по менеджерам, visual alerts.

### Сценарий 7: Менеджер уехал в отпуск
Его сделки нужно передать → руководитель массово переназначает менеджера → новый менеджер читает историю контактов и продолжает с того места.

**Что нужно:** хорошая история коммуникаций, массовое переназначение.

### Сценарий 8: Дубль заявки
Гость подал заявку дважды (с другого устройства или забыл) → система должна распознать и не создавать вторую сделку.

**Что нужно:** проверка по телефону + email при создании, предупреждение менеджеру.

### Сценарий 9: Передача на расселение
Менеджер по продажам довёл гостя до оплаты → теперь нужно забронировать конкретную комнату → это делает менеджер по расселению (другой человек). Менеджер по продажам нажимает «Готов к расселению» → менеджер по расселению видит запрос на своей странице (placement/) → назначает комнату через шахматку → в карточке сделки автоматически появляется: «Комната 305, Гостевой дом, 22 авг – 6 сен».

Менеджер по продажам может потом сообщить гостю: «Для вас забронирована комната 305 в Гостевом доме».

**Что нужно:** связь между crm_deals и residents (через vaishnava_id + retreat_id), отображение размещения в карточке сделки, сигнал менеджеру по расселению.

### Сценарий 10: Гость оплатил, но комнату ещё не дали
Менеджер по продажам видит в чеклисте «Проживание: В ШРСК», но в блоке «Размещение» пусто — комната не назначена. Он может нажать «Запросить расселение», чтобы создать задачу для менеджера по расселению.

**Что нужно:** блок «Размещение» в карточке с реальными данными из таблицы residents/rooms.

## Страница: Карточка сделки (deal.html)

Главный рабочий инструмент. Менеджер проводит здесь 80% времени.

### Верхняя полоса (статус-баннер)
**Текущее состояние:** цветная полоса со статусом и именем гостя.

**Улучшения:**
- Добавить номер телефона и email прямо в баннер (кликабельные — копируются в буфер)
- Кнопки быстрого действия: «Позвонить» (tel: ссылка), «WhatsApp» (wa.me/ ссылка), «Telegram» (t.me/ ссылка)
- Индикатор «Повторный гость» — маленький бейдж рядом с именем, если есть прошлые сделки

### Боковая панель (правая колонка)

Порядок секций сверху вниз — по частоте использования:

#### 1. Статус и режим работы
**Как сейчас:** два селекта (статус + режим работы).
**Оставить как есть** — работает хорошо.

#### 2. Менеджер
**Как сейчас:** селект с выбором менеджера.
**Добавить:** кнопку «Взять себе» (один клик — назначает текущего пользователя).

#### 3. Записать контакт (quick log)
**Как сейчас:** 5 кнопок-типов + textarea + «Записать».
**Улучшения:**
- После записи показать последние 3 записи прямо под формой (свернуть в accordion)
- Формат: `[иконка типа] [время] — [текст]`
- Счётчик коммуникаций: «Контактов: 5»
- Это снимает необходимость переключаться на вкладку «История» для проверки

#### 4. Чеклист гостя (НОВЫЙ блок)
**Зачем:** менеджер видит, что уже выяснил, а что нет.

Визуально — компактная сетка 2×3 с иконками:

```
🏠 Проживание    [В ШРСК ▾]
🍽️ Питание       [В ШРСК ▾]
✈️ Прибытие      [Нужен трансфер ▾]
✈️ Отъезд        [Не нужен ▾]
📋 Виза          [Не нужна ▾]
🎫 Билеты        [Куплены ▾]
```

Каждый пункт — компактный селект. Значения:

| Поле | Варианты | По умолчанию |
|------|----------|--------------|
| Проживание | В ШРСК / Партнёрская гостиница / Самостоятельно / Не выяснено | Не выяснено |
| Питание | В ШРСК / Самостоятельно / Не выяснено | Не выяснено |
| Трансфер прибытие | Нужен / Не нужен / Не выяснено | Не выяснено |
| Трансфер отъезд | Нужен / Не нужен / Не выяснено | Не выяснено |
| Виза | Нужна помощь / Не нужна / Не выяснено | Не выяснено |
| Билеты | Куплены / Не куплены / Не нужны / Не выяснено | Не выяснено |

Сохраняется автоматически при изменении (без кнопки «Сохранить»).
«Не выяснено» — серый текст. Остальные — обычный.

#### 5. Данные рейсов (НОВЫЙ блок)

Появляется когда «Трансфер прибытие» или «Трансфер отъезд» ≠ «Не выяснено».

```
✈️ Прилёт
  Рейс: AI 332         Дата: 21.08.2026 15:45
  Аэропорт: DEL        Трансфер: ✅ нужен
  Ранний заезд: ✅ (прилёт до 10:00)

✈️ Вылет
  Рейс: AI 331         Дата: 07.09.2026 23:10
  Аэропорт: DEL        Трансфер: ✅ нужен
  Поздний выезд: ✅ (вылет после 20:00)
```

**Поля:**
- `arrival_type` — тип прибытия: `flight` (самолёт) / `self` (самостоятельно, уже в Индии) / `other` (поезд, автобус и т.д.)
- `arrival_flight` — номер рейса прилёта (текст, только если type=flight)
- `arrival_datetime` — дата/время прибытия (TIMESTAMPTZ) — прилёта ИЛИ приезда
- `arrival_airport` — аэропорт (DEL / AGR / LKO / другой, только если type=flight)
- `arrival_needs_transfer` — берётся из чеклиста
- `arrival_early_checkin` — ранний заезд (bool)
- `departure_type` — тип отъезда: `flight` / `self` / `other`
- `departure_flight` — номер рейса вылета (только если type=flight)
- `departure_datetime` — дата/время отъезда
- `departure_airport` — аэропорт (только если type=flight)
- `departure_needs_transfer` — из чеклиста
- `departure_late_checkout` — поздний выезд (bool)

**Варианты прибытия:**
1. **Самолёт** (`flight`) → показываем поля: рейс, аэропорт, время. Авто-расчёт: приезд в ШРСК = прилёт + 4ч
2. **Самостоятельно** (`self`) → только дата/время приезда в ШРСК (гость сообщает напрямую)
3. **Другой транспорт** (`other`) → дата/время + текстовое поле «откуда» (поезд из Дели, автобус и т.д.)

Аналогично для отъезда:
1. **Самолёт** → рейс, аэропорт, время. Авто-расчёт: выезд из ШРСК = вылет − 6ч
2. **Самостоятельно** → дата/время выезда из ШРСК
3. **Другой транспорт** → дата/время + «куда»

**Автоматика раннего заезда / позднего выезда:**
- Если `arrival_type = flight`: приезд в ШРСК = прилёт + 4 часа. Если **до 10:00** → ранний заезд
- Если `arrival_type = self/other`: берём `arrival_datetime` как есть. Если **до 10:00** → ранний заезд
- Если `departure_type = flight`: выезд из ШРСК = вылет − 6 часов. Если **после 13:00** → поздний выезд
- Если `departure_type = self/other`: берём `departure_datetime` как есть. Если **после 13:00** → поздний выезд
- Менеджер может переключить вручную

**Важно:** эти данные нужны для формирования трансферного листа (`guest_transfers`) — при передаче на расселение данные рейсов автоматически попадают в систему трансферов.

#### 6. Дополнительные поля гостя (НОВЫЙ блок)

Выясняются в процессе переговоров, не обязательные:

```
👤 Дополнительно
  Духовный учитель: [                    ]
  Хочет жить рядом: [                    ]
  Особые потребности: [                  ]
```

**Поля (в таблице `vaishnavas`):**
- `guru_name` — имя духовного учителя (текст)

**Поля (в таблице `crm_deals`):**
- `roommate_preference` — с кем хочет жить рядом (текст, свободный ввод)
- `special_needs` — особые потребности: диета, здоровье, мобильность (текст)

Эти данные передаются менеджеру по расселению и помогают разместить гостя комфортно.

#### 7. Индикатор «Готовность к передаче» (НОВЫЙ)

Блок вверху боковой панели, показывающий менеджеру — готов ли гость к передаче на расселение.

```
📋 Готовность к передаче
  ✅ Оплата         ✅ Проживание
  ✅ Питание         ✅ Прилёт
  ❌ Вылет           ✅ Виза
  ───────────────────────
  Готово 5 из 6
  [Передать на расселение]
```

**Логика:**
- Каждый пункт чеклиста + данные рейсов → зелёный если заполнен, красный если нет
- Когда все пункты зелёные (или «не нужно») → кнопка «Передать на расселение» становится активной
- Кнопка создаёт задачу для менеджера по расселению со всей собранной информацией
- Неполная передача возможна, но система предупредит: «Не заполнен вылет. Передать всё равно?»

**Хранение:** новые колонки в `crm_deals`:
```sql
checklist_accommodation TEXT CHECK (... IN ('ashram','partner','self','unknown')) DEFAULT 'unknown'
checklist_meals TEXT CHECK (... IN ('ashram','self','unknown')) DEFAULT 'unknown'
checklist_transfer_arrival TEXT CHECK (... IN ('needed','not_needed','unknown')) DEFAULT 'unknown'
checklist_transfer_departure TEXT CHECK (... IN ('needed','not_needed','unknown')) DEFAULT 'unknown'
checklist_visa TEXT CHECK (... IN ('need_help','not_needed','unknown')) DEFAULT 'unknown'
checklist_tickets TEXT CHECK (... IN ('bought','not_bought','not_needed','unknown')) DEFAULT 'unknown'
```

#### 5. Финансы
**Как сейчас:** 4 блока (начислено, оплачено, баланс, депозит).
**Оставить** — работает хорошо. Цветовая индикация баланса (красный/зелёный/синий) очень удобна.

#### 6. Действия
**Как сейчас:** шаблоны, отмена, удаление.
**Улучшения:**
- «Удалить» — показывать только для статуса «Отменена»
- При отмене — модалка с обязательным выбором причины:
  - Передумал
  - Финансовые причины
  - Проблемы с визой
  - Проблемы со здоровьем
  - Перенёс на другой ретрит
  - Не отвечает
  - Другое (текстовое поле)

### Вкладка «Инфо»

**Контактные данные** (левая колонка):
- Телефон (кликабельный → копировать)
- Email (кликабельный → копировать)
- Telegram (кликабельный → открыть t.me/)
- Город
- Ссылка «Открыть профиль →»

**Ретрит и условия** (правая колонка):
- Ретрит (бейдж с цветом)
- Даты ретрита
- Источник заявки
- Дата создания

**Состав группы** (НОВЫЙ блок, после контактных данных):
```
Участники (3):
  👤 Рамачандра дас — основной контакт
  👤 Сита деви — жена
  👶 Лакшми — дочь, 8 лет
  [+ Добавить участника]
```

Данные участника: имя, связь (жена/муж/ребёнок/друг/коллега), возраст (опционально, для детей).

**Хранение:** новая таблица `crm_deal_members`:
```sql
CREATE TABLE crm_deal_members (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    deal_id UUID REFERENCES crm_deals(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    relation TEXT, -- wife/husband/child/friend/colleague
    age INTEGER, -- для детей
    notes TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
);
```

**История гостя** (блок под составом группы):
```
📋 Был раньше (2 ретрита):
  🟢 Лила-киртан 2025 (фев 2025) — Завершена, ₹15 000
  🟡 Бхагаватам 2024 (авг 2024) — Счёт отправлен (не оплатил)
```
- Зелёный = completed/fully_paid/checked_in
- Жёлтый = lead/contacted/invoice_sent/prepaid (не дошёл)
- Серый = cancelled
- Клик → открывает ту сделку

**Размещение** (НОВЫЙ блок, после истории гостя):

Данные подтягиваются автоматически из таблицы `residents` по связке `vaishnava_id + retreat_id`.

Три состояния:

1. **Комната не назначена, чеклист «Проживание» = не выяснено:**
   ```
   🏠 Размещение
   Не определено
   ```

2. **Чеклист «Проживание» = В ШРСК, но комната не назначена:**
   ```
   🏠 Размещение: в ШРСК
   Комната не назначена
   [Запросить расселение]
   ```
   Кнопка «Запросить расселение» создаёт задачу для менеджера по расселению:
   - Заголовок: «Расселить [имя гостя] на [ретрит]»
   - Назначается на пользователя с правом `edit_placement`
   - Приоритет: normal (high если ретрит начинается через <7 дней)
   - В описании: предпочтения гостя, состав группы, заметки

3. **Комната назначена (есть запись в `residents`):**
   ```
   🏠 Размещение: Гостевой дом, комната 305
   22 авг – 6 сен 2026
   [Открыть в шахматке →]
   ```
   Ссылка ведёт на `placement/timeline.html` с фокусом на этой комнате.

**Запрос данных:**
```sql
SELECT r.check_in, r.check_out, r.status,
       rooms.number, buildings.name_ru
FROM residents r
JOIN rooms ON r.room_id = rooms.id
JOIN buildings ON rooms.building_id = buildings.id
WHERE r.vaishnava_id = deal.vaishnava_id
  AND r.retreat_id = deal.retreat_id
```

Если записей нет — показываем «Комната не назначена». Менеджеру по продажам не нужно идти в модуль размещения — он видит результат прямо в карточке сделки.

**Заметки** (внизу):
- Текст из `deal.notes`
- Кнопка «Редактировать» → textarea → «Сохранить»

**Улучшения:**

## Страница: Канбан (index.html)

Основной обзор для руководителя и быстрый доступ для менеджера.

### Текущее состояние
Работает хорошо: drag-n-drop, фильтры, real-time.

### Улучшения

#### Счётчики на колонках
Каждая колонка показывает:
- Количество сделок
- Сумму оплат (если > 0)

```
Новая заявка (5)        Контакт (12)        Счёт отправлен (3)
                         ₹45 000              ₹78 000
```

#### Визуальные сигналы тревоги
- Колонка «Новая заявка» с >3 карточками → красный фон заголовка
- Карточки без контакта >24ч → красная левая полоска
- Карточки со статусом «Счёт отправлен» >5 дней → оранжевая полоска

#### Информация на карточке
Текущее: имя, телефон, ретрит, оплачено, менеджер.

Добавить:
- Иконки чеклиста: 🏠✈️🍽️ — маленькие, серые (не выяснено) или цветные (заполнено)
- Дата последнего контакта: «3 дня назад» (текст, если >2 дней — оранжевый)

#### Быстрые действия
- Наведение на карточку → кнопка «→» (перевести на следующий статус без открытия)
- Клик по телефону → копировать в буфер

#### Фильтр по ретриту — табами
Вместо селекта → горизонтальные табы: «Все» | «Сева-ретрит» | «Ретрит АБ» | ...
Активный таб подчёркнут цветом ретрита.

## Страница: Таблица сделок (deals.html)

Для поиска конкретной сделки и массовых операций.

### Текущее состояние
Работает хорошо: таблица, фильтры, пагинация, создание сделки.

### Улучшения

#### Колонка «Чеклист»
Между колонкой «Статус» и «Менеджер» добавить колонку с иконками:
```
🏠 ✈️ 🍽️ 📋 🎫
```
Серые = не выяснено, цветные = заполнено. Менеджер в таблице видит, у кого чеклист пустой.

#### Массовые действия
- Чекбоксы в первой колонке
- При выборе >=1 строки → панель действий сверху:
  - «Назначить менеджера» → селект → применить ко всем выбранным
  - «Сменить статус» → селект → применить
  - «Удалить» (только для отменённых)

#### Экспорт
- Кнопка «Скачать .xlsx» (рядом с «Новая сделка»)
- Экспортирует с текущими фильтрами
- Колонки: имя, телефон, email, ретрит, статус, менеджер, оплачено, дата

#### Создание сделки — улучшение модалки
Текущее: при выборе гостя не видно его данных.

Добавить: после выбора гостя показать превью:
```
👤 Рамачандра дас (Тестимя Тестфио)
📞 +7 999 123 45 67
✉️ test@email.com
```
Чтобы менеджер убедился, что выбрал правильного человека.

## Страница: Публичная форма (form.html)

### Текущее состояние
Работает: одна страница, 6 полей + сообщение.

### Улучшения

#### Защита от дублей
При отправке — перед созданием сделки проверить:
```sql
SELECT id FROM crm_deals
WHERE retreat_id = ? AND vaishnava_id = ? AND status != 'cancelled'
```
Если найдена → показать: «Вы уже зарегистрированы на этот ретрит. Если у вас есть вопросы, напишите нам: [контакт]»

#### Мультиязычность
- URL: `form.html?retreat=X&lang=en`

- Все лейблы, плейсхолдеры, сообщения об ошибках — на двух языках

- Кнопка «Вернуться на сайт» → rupaseva.com

## Страница: Шаблоны сообщений — НОВАЯ (templates.html)

### Зачем
Менеджер не должен каждый раз придумывать текст. Один клик — шаблон скопирован, можно вставить в WhatsApp.

### Список шаблонов

| Шаблон | Когда использовать |
|--------|-------------------|
| Первый контакт | После получения заявки |
| Реквизиты для оплаты | При отправке счёта |
| Напоминание об оплате | Через 5 дней после счёта |
| Подтверждение оплаты | После получения оплаты |
| Информация о трансфере | После покупки билетов |
| Подготовка к ретриту | За неделю до начала |
| Благодарность | После завершения ретрита |

### Переменные в шаблонах
```
{имя} — имя гостя (spiritual_name или first_name)
{ретрит} — название ретрита
{дата_начала} — дата начала ретрита
{дата_окончания} — дата окончания
{сумма} — сумма к оплате
{оплачено} — уже оплачено
{баланс} — остаток к оплате
```

### Интерфейс
- Таблица шаблонов с кнопкой «Редактировать»
- В карточке сделки → кнопка «Шаблоны» → модалка со списком → клик → переменные заполняются данными текущей сделки → «Скопировать»

### Миграция 144: Состав группы
```sql
CREATE TABLE crm_deal_members (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    deal_id UUID NOT NULL REFERENCES crm_deals(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    relation TEXT,
    age INTEGER,
    notes TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
);

-- RLS
ALTER TABLE crm_deal_members ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Authenticated read deal members" ON crm_deal_members
  FOR ALL TO authenticated USING (true);

-- Анон: insert для публичной формы (если будем добавлять состав)
CREATE POLICY "Anon insert deal members" ON crm_deal_members
  FOR INSERT TO anon WITH CHECK (true);
```

## Приоритеты реализации

### Фаза 1 — Основное (неделя 1)
> Без этого менеджер не может нормально работать.

1. **Чеклист гостя** — миграция + UI в боковой панели deal.html
2. **Состав группы** — миграция + UI на вкладке «Инфо»
3. **История гостя** — запрос прошлых сделок на вкладке «Инфо»
4. **Блок «Размещение»** — данные из residents/rooms на вкладке «Инфо» + кнопка «Запросить расселение»
5. **Причина отмены** — модалка при отмене + поле в БД
6. **Удаление отменённых** — кнопка только для cancelled
7. **Защита от дублей** — проверка в form.html перед insert
8. **Быстрые контакты в баннере** — кликабельные телефон/WhatsApp/Telegram

### Фаза 2 — Удобство (неделя 2)
> Ускоряет работу менеджера в 2 раза.

9. **Последние контакты в сайдбаре** — 3 записи под формой «Записать контакт»
10. **Счётчики на канбане** — количество + суммы на колонках
11. **Чеклист в таблице сделок** — иконки-индикаторы
11. **Шаблоны сообщений** — таблица + модалка в deal.html
12. **Автокурс валют** — кнопка + API

### Фаза 3 — Контроль (неделя 3)
> Для руководителя.

13. **Аналитика** — дашборд по ретриту + воронка + по менеджерам
14. **Массовые действия** — чекбоксы + панель в deals.html
15. **Экспорт в Excel** — кнопка в deals.html
16. **Быстрые действия на канбане** — кнопка «→» на карточке

### Фаза 4 — Автоматизация (неделя 4)
> Система работает за менеджера.

17. **Автозадачи** — триггеры/Edge Functions
18. **Мультиязычная форма** — EN-версия form.html
19. **Бейдж задач в меню** — счётчик активных задач
20. **Уведомления** — утренняя сводка в Telegram
