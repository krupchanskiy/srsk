### G2 — cutover-симуляция (K1, BEGIN/ROLLBACK)
Сценарии: 28, 49, 65. В транзакции: reset-тело → opening счетов → load_opening_balances
(долг general, долг accommodation, аванс credit) → проверка: балансы участников верны,
«всего под ответственностью» = сумма opening без задвоения, повторная загрузка → skip,
изменённая строка → конфликт всей пачки.

### G3 — роль account_user (K1)
Сценарий 93 + матрица прав: выдать тест-пользователю `fin_account_user` + `fin_account_access`
(в rollback), проверить: его expense по своему счёту → pending; по закрытому ретриту →
`object_closed`; чужой счёт → forbidden.

## 5. Гэпы реализации — решить ДО запуска программы

1. **Сценарий 68** — «правка `payer_contact_id` платежа без активного refund → разрешено
   под блокировкой операции с аудитом». RPC для этого нет (update_posting_analytics правит
   проводку, payer живёт на операции). Варианты: (а) маленькая RPC `fin_update_payer`
   (~40 строк, лок операции + net_refunded=0 + аудит); (б) отложить в бэклог с фиксацией
   в памяти — операционный обход: сторно + повторный ввод. **Рекомендую (а)** — дёшево.
2. **Сценарий 89** — вложение проводки счёта Кафе видно пользователю с доступом к Кафе.
   `fin_v_attachments` сейчас admin/observer-only. Экранов департаментов ещё нет, но
   view поправить дёшево: ветка `EXISTS (fin_account_access через posting_id)`.
   **Рекомендую сделать** вместе с (1) одной миграцией 200.
