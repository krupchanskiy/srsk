# UX-аналитика рабочих сценариев финмодуля — v1.0

**Дата:** 20.07.2026. **Статус:** аналитика для обсуждения, не реализовано.
**Метод:** каждая страница разобрана через реальные рабочие циклы из ТЗ v4.2.10,
cutover-runbook и приёмочных сценариев — не «что было бы красиво», а «что человек
делает на этой странице десять раз в день и где он спотыкается».

## 1. Главная (index.html)

**Кейсы:** утренний вход «всё ли в порядке» → быстрый переход к работе.

| # | Предложение | Зачем (кейс) | Сложн. |
|---|---|---|---|
| 1.1 🔥 | **Карточка «Давность сверки»**: «Касса сверена сегодня ✓ / 3 дня назад ⚠» по каждому реальному счёту, клик → сверка с предвыбранным счётом | Ежедневный ритуал — ядро запуска (3 чекпоинта подряд). Сейчас о несверенности ничто не напоминает | S |
| 1.2 🔥 | **Минусы кликабельны**: клик по счёту в «Минусах» → лента этого счёта в ДДС | Увидел минус → первый вопрос «откуда?» — сейчас ищешь руками | XS |
| 1.3 ⭐ | **Быстрые действия** на главной: «+ Расход», «+ Платёж», «+ Перевод» | Самые частые операции доступны без промежуточного шага | S |
| 1.4 ⭐ | **«Требует закрытия»**: ретрит закончился N дней назад, а закрытия нет / отчёт dirty — строка с кнопкой | Из интеграционного исследования (И3); закрытия забываются | M |

## 2. ДДС (dds.html) — главный журнал

**Кейсы:** «найди платёж на 5 000 от вторника»; дублирование CRM-платежа; регулярный расход («опять за газ, как в прошлый раз»); shadow-сверка выборки с Excel.

| # | Предложение | Зачем (кейс) | Сложн. |
|---|---|---|---|
| 2.1 🔥 | **Поиск по сумме**: ввёл «5000» в поиск — нашлись операции с такой суммой (в любой валюте) | Сумма — главный идентификатор операции в голове человека; сейчас поиск только по тексту | S |
| 2.2 🔥 | **Пресеты периода**: Сегодня · Неделя · Месяц · Ретрит (чипы над фильтрами) | Ручной ввод двух дат — самая частая микро-боль журнала | S |
| 2.3 🔥 | **Итог по выборке**: под таблицей «Приход ₹X · Расход ₹Y» по отфильтрованному | Shadow-сверка с Excel: «сколько всего расходов за вчера?» — сейчас складываешь глазами | S |
| 2.4 ⭐ | **«Повторить» из разворота операции**: открывает форму расхода/прихода, предзаполненную по образцу (счёт, статья, СС, контрагент — всё кроме суммы/даты) | Регулярные расходы (газ, овощи, зарплаты) вводятся ежедневно одинаково; и дублирование CRM-платежей быстрее | M |
| 2.5 ⭐ | **Фильтр по статье** (категории) | «Сколько мы потратили на транспорт» — сейчас нельзя без Аналитики, а там нет детализации по операциям | S |
| 2.6 ⭐ | **Deep-link на операцию** `?op=<id>`: автоскролл + разворот + подсветка | Ссылка на операцию в WhatsApp/задаче — «посмотри вот эту» | S |
| 2.7 ⭐ | **Экспорт CSV текущей выборки** | Shadow: ежедневное сравнение с Excel — идеально отдать выборку в файл | S |
| 2.8 💡 | Автор операции (кто создал) в развороте | Разбор спорных: «кто это ввёл?» | XS |
| 2.9 💡 | Единая кнопка «+ Операция» с выбором типа (включая «Платёж участника» — переход к Участникам с открытой формой) | Из критики: типы операций разнесены по страницам по внутренней модели БД | M |

## 3. Участники (participants.html)

**Кейсы:** день заезда — очередь гостей, «сколько я должен?»; сообщить гостю сводку в WhatsApp; собрать долги перед закрытием; групповой платёж «муж платит за семью» (сц. 3 ТЗ).

| # | Предложение | Зачем (кейс) | Сложн. |
|---|---|---|---|
| 3.1 🔥 | **Поиск по имени** над списком участников | На ретрите 50–150 человек; в день заезда это очередь живых людей | S |
| 3.2 🔥 | **Фильтр-чипы: Все · Должники · Авансы** + итоговая строка (сумма долгов/авансов, счётчики) | Перед закрытием работа идёт только с должниками; итог долга ретрита — постоянный вопрос ВГ | S |
| 3.3 ⭐ | **«Скопировать сводку» в карточке**: кнопка формирует текст «Оргвзнос: 15 000, оплачено 10 000. Долг: ₹ 5 000» → в буфер | Гостям сводку шлют в WhatsApp — сейчас перепечатывают вручную | S |
| 3.4 ⭐ | **Групповой платёж — участник в строке платежа** (сц. 3 ТЗ: «групповой платёж за троих → три проводки»). Сервер это умеет, UI — нет: сейчас все строки платежа привязаны к плательщику | Семьи и группы платят одним переводом за нескольких — функциональный гэп UI против приёмочного сценария | M |
| 3.5 💡 | Из карточки участника → его операции в ДДС (deep-link с фильтром) | «Покажи все его платежи целиком» | S |
| 3.6 💡 | Сортировка списка кликом по заголовку (по долгу ↓) | Быстро увидеть крупнейших должников | S |

## 4. Счета (accounts.html)

**Кейсы:** «что происходило по кассе?»; выдать подотчёт → принять остаток; проверить давность сверки.

| # | Предложение | Зачем (кейс) | Сложн. |
|---|---|---|---|
| 4.1 🔥 | **Клик по строке счёта → лента счёта в ДДС** (`dds.html?account=`) | Самое естественное желание на этой странице — сейчас строка некликабельна, есть только правка | XS |
| 4.2 🔥 | **Кнопка «Сверить» в строке** → сверка с предвыбранным счётом (`reconciliation.html?account=`) | Убирает два клика из ежедневного ритуала | S |
| 4.3 ⭐ | **Цвет давности чекпоинта**: сегодня — зелёный, 2–3 дня — жёлтый, дольше/никогда — красный | Мгновенно видно, где ритуал сломался | XS |
| 4.4 💡 | «Принять возврат» рядом с «Выдать» на подотчётных счетах (перевод custodial→real с предвыбором) | Вторая половина цикла подотчёта (сц. 8) — сейчас только через общий перевод | S |

**Кейсы:** ежедневный пересчёт кассы по номиналам — руки на цифровом блоке; разбор прошлого расхождения.

| # | Предложение | Зачем (кейс) | Сложн. |
|---|---|---|---|
| 6.1 🔥 | **Deep-link `?account=`** (для 1.1 и 4.2) + автофокус на первом номинале | Вход в ритуал одним кликом откуда угодно | XS |
| 6.2 ⭐ | **Клавиатурный поток по номиналам**: Enter → следующее поле, значение выделяется при фокусе | Пересчёт — это ввод 7–14 чисел подряд; мышь тут лишняя | S |
| 6.3 ⭐ | **Success-состояние после сверки**: «Сверено ✓ №26» крупно + счётчик серии («2 чекпоинта подряд без расхождений из 3») | Peak-end ежедневного ритуала + геймификация критерия запуска из runbook §4 | S |
| 6.4 💡 | Клик по строке истории → детали пересчёта (какие номиналы, какие точки) | Разбор вчерашнего расхождения | M |

| # | Предложение | Зачем (кейс) | Сложн. |
|---|---|---|---|

## Топ приоритетов (что даёт максимум пользы на рубль)

**Пакет A «Ежедневный ритуал»** (готовит запуск): 1.1 давность сверки на главной → 4.2/4.3 сверить из счетов + цвет давности → 6.1/6.2/6.3 deep-link, клавиатура номиналов, success-серия. Итог: цикл «зашёл → увидел → сверил → серия чекпоинтов» без единого лишнего клика.

**Пакет B «Журнал, которому доверяют»**: 2.1 поиск по сумме → 2.2 пресеты периодов → 2.3 итог по выборке → 2.6 deep-link → 2.7 CSV. Прямо обслуживает shadow-сверку с Excel.

**Пакет C «День заезда»**: 3.1 поиск по имени → 3.2 фильтры должников + итог → 3.3 копировать сводку → 3.4 групповой платёж (закрывает сц. 3 ТЗ).

**Пакет D «Cutover»** (до даты X): 8.1 поиск в справочниках → 8.2 массовый импорт статей.

Пакеты A–D почти не пересекаются кодом и делаются независимо, каждый — примерно день работы.
