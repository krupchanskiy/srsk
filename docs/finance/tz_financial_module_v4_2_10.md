# ТЗ: Финансовый модуль бэк-офиса ШРСК

**Версия:** 4.2.10
**Автор:** Сварупананда дас
**Дата:** июль 2026
**Статус:** на согласовании (АК Прабху, Ванамали Гопал Прабху, Нарада Прабху)

**Описание:** финансовый модуль бэк-офиса ШРСК — учёт денег от счетов (реальных и подотчётных), начисления и платежи участников ретритов, сверка остатков, закрытие ретритов с версионируемым итоговым отчётом. Деньги неизменны и правятся только сторно; аналитика правится с аудит-следом; редкие кейсы решает человек по зафиксированному правилу, а не универсальный механизм. Модель данных: 16 таблиц, 7 инвариантов.

---

## 1. Контекст и цели

### Принцип соразмерности

Система строится ради доверия к цифрам, но соразмерно масштабу: тысячи операций в год, 2–3 доверенных администратора, один разработчик. Строгость — там, где ошибка необнаружима (**деньги**: суммы, счета, даты). Гибкость — там, где ошибку видно и легко исправить (**аналитика**: статьи, привязки к ретритам). Редкие кейсы решает человек по зафиксированному правилу (раздел 8), а не механизм.

### Текущее состояние

Финансовый учёт ШРСК ведёт один человек — Ванамали Гопал Прабху — в Excel-таблицах:

- **Закрытость** — таблицы понятны только автору, проверить результат практически невозможно
- **Задержка** — итоги появляются после ретритного сезона, а не в реальном времени
- **Хрупкость** — формулы проверяются вручную, всё завязано на одного человека
- **Ограниченность** — учёт построен вокруг ретритов; зарплаты, электричество, гостиница вне ретритов остаются за кадром
- **Нет делегирования** — департаменты не вносят отчёты сами: приносят бумажки и сообщения в Telegram, Ванамали Гопал переносит вручную

Кассовая дисциплина при этом на месте: деньги выдаёт только Ванамали Гопал или уполномоченное лицо. Система оформляет существующую дисциплину, а не создаёт её.

### Целевое состояние

Финансовый модуль в составе бэк-офиса ШРСК (in.rupaseva.com):

- Финансовая картина в реальном времени: остатки, долги, траты департаментов
- Департаменты вносят траты сами; администратор согласовывает по документам
- Полная замена Excel после даты отсечения — ни одна табличка не остаётся вне системы
- Проверяемость: денежная история не переписывается; каждое изменение оставляет след
- Guest Portal показывает участнику его счёт — как в отеле

### Принципиальные ограничения

- **Нет интеграции с банками** — ручной ввод из выписок, ежедневный ритуал
- **Нет ретроспективы** — на дату запуска вносятся текущие остатки и открытые позиции; история Excel не мигрирует
- **CRM-интеграция — фаза 2** — до неё платежи по ретритам, проведённые через CRM, дублируются в финмодуль вручную (осознанное решение: объём посилен, зато MVP не зависит от интеграции)
- **CAPEX вне системы** — стройку АК ведёт отдельно
- **Номенклатура мероприятий — вне финмодуля** — финмодуль ссылается на учётные объекты (в MVP это только ретриты); события гестхауса и сторонние мероприятия появятся как новые типы объекта того же справочника, без миграции существующих данных (см. 4.13)

## 2. Глоссарий

| Термин | Определение |
|--------|-------------|
| **Счёт** | Место, где лежат деньги, в одной валюте. Реальный или подотчётный |
| **Реальный счёт** | Физическая точка денег организации: касса ₹, карта ШРСК, PayPal |
| **Подотчётный счёт** (custodial) | Деньги организации, физически находящиеся на руках у ответственного лица — департамента (Кафе) или внешнего подотчётника (Ашиш). Остаток счёта = за что человек материально отвечает |
| **Группа счетов** | Текстовая метка, объединяющая валютные счета одной точки: «Касса» = Касса/₹ + Касса/₽ + Касса/$ |
| **Операция** | Одно бизнес-действие: платёж, выдача, трата, перевод, сторно, корректировка сверки |
| **Проводка** | Денежная строка операции: счёт + направление + сумма + аналитика (статья, объект учёта, участник) + порядковый номер в ленте счёта (`ledger_seq`) |
| **Статья** | Категория дохода или расхода из справочника |
| **Cost center** | Подразделение-бенефициар на проводке — для трат одного департамента за другой. Значение из справочника (`fin_cost_centers`), не свободный текст |
| **Начисление** | Обязательство участника заплатить (оргвзнос, проживание, питание) со снапшотом цены на момент создания, только в ₹ |
| **Долг / аванс участника** | Разница начислений и signed платежей участника по блоку (оргвзнос/проживание/питание/дополнительно/общее) + неденежная входящая позиция на дату запуска |
| **Согласование** | Подтверждение операции департамента администратором по документам. На деньги не влияет. Статусы: не проверено / подтверждено / оспорено |
| **Оспорено (disputed)** | Администратор не согласен с тратой департамента (чек не принят, назначение спорно), но деньги физически не возвращены. Не создаёт сторно; решение — по правилу раздела 8 |
| **Сторно** | Операция-зеркало с обратными проводками, ссылкой на исходную и причиной. Единственный способ исправить деньги задним числом. Всегда полное; частичной отмены нет |
| **Сверка** | Сопоставление расчётного остатка счёта с фактическим (пересчёт наличных / выписка) |
| **Чекпоинт** | Подтверждённая сверка: «все проводки счёта до номера `cutoff_ledger_seq` совпадают с реальностью» — а не «на эту дату» |
| **Учётный объект** | Сущность, к которой привязана проводка вместо прямой ссылки на ретрит: ретрит сегодня, в фазе 2 — период гестхауса, стороннее событие (`fin_accounting_objects`) |
| **Точка хранения** | Физическое место наличных внутри кассы: на руках, сейф. Учитывается только при сверке |
| **Выемка бокса** | Пересчёт содержимого бокса пожертвований + приход в кассу. До выемки содержимое бокса вне учёта |
| **Закрытие ретрита** | Финансовое завершение: первая (`is_initial`) версия в `fin_object_closures` с итоговым снимком цифр; PDF формируется отдельным шагом. Новые операции запрещены, кроме ограниченного набора post-close; такая операция или правка начисления не создаёт новую версию сразу — только помечает отчёт устаревшим (`report_dirty_at`), следующая версия появляется по кнопке «Перевыпустить» |

## 3. Роли и доступ

### Администратор

**Кто:** Ванамали Гопал, АК, резервные администраторы кассы (на время отъезда основного). Несколько одновременно — все действия различимы в аудит-логе, отдельные роли и параллельные журналы не нужны.

**Может:** всё — операции любых типов, согласование, сторно, сверки, закрытие ретритов, справочники, начисления, назначение доступа к счетам.

### Наблюдатель

**Кто:** попечители, бухгалтер при подготовке внешней отчётности.

**Может:** видеть все экраны, ДДС, аналитику, аудит-лог. Ноль кнопок записи.

### Пользователь счетов

**Кто:** руководители департаментов (Джамбала-маллика — кафе, Сундара Рупа — кухня), главы направлений (Премамрита Прабху — поклонение, несколько счетов), внешние подотчётники с доступом (Ашиш).

**Может:**
- Видеть свои счета (назначенный набор в `fin_account_access`): остаток, историю операций
- Вносить траты по своим счетам (короткий список статей или «Прочее»)
- Вносить трату за другой департамент (пометка cost center; видит названия чужих счетов, не балансы)
- Переводить деньги на другой подотчётный счёт

**Не может:** видеть чужие балансы и операции, общий ДДС, аналитику, справочники; создавать статьи.

### Участник ретрита

**Кто:** гости, через Guest Portal. Видит своё сальдо: начислено / оплачено / долг или аванс, детализацию платежей.

## 4. Модель данных (16 таблиц)

### 4.1 fin_accounts — счета

| Поле | Тип | Описание |
|------|-----|----------|
| id | uuid PK | |
| name | text | «Касса ₹», «Кафе», «Ашиш» |
| kind | enum | `real` / `custodial` |
| reconciliation_mode | enum | `cash_count` / `statement`. Определяет протокол `perform_reconciliation` (раздел 5, 10): `cash_count` — сервер сам вычисляет `counted_balance` из `counts`, `statement_balance` не принимается; `statement` — клиент передаёт `statement_balance` напрямую, `counts` обязан быть `NULL`. Не выводится из `kind` или названия счёта (и касса, и карта, и PayPal технически `real`, но сверяются по-разному) — независимое поле, задаётся при создании счёта |
| currency | FK → fin_currencies | Одна валюта; не меняется после первой проводки |
| group_name | text | «Касса», «PayPal», «Поклонение» — метка для группировки в UI |
| responsible_person | FK → contacts | Материально ответственный |
| default_cost_center_id | FK → fin_cost_centers | Для custodial-счетов департаментов — свой пункт справочника (например, счёт «Кухня/₹» → «Кухня»). Используется **только как подсказка при заполнении формы траты** — RPC записывает выбранное значение прямо в `fin_postings.cost_center_id` при создании проводки. Аналитика никогда не обращается к текущему значению счёта повторно — иначе смена значения задним числом исказила бы уже проведённые проводки. Для реальных счетов — null |
| is_active | bool | Деактивация — только при нулевом остатке |
| created_at | timestamptz | |

**Реальные счета** (перечень уточняет Ванамали Гопал): Касса/₹, Касса/₽, Касса/$, Касса/€ (одна физическая коробка — несколько счетов), Карта ШРСК/₽, PayPal/$, PayPal/€.

**Подотчётные счета:** департаменты — Кафе/₹, Кухня/₹, Гестхаус/₹, Зелёные/₹, Магазин/₹; направление поклонения дробится: Поклонение-швеи/₹, Поклонение-украшения/₹, Поклонение-пуджари/₹ (Премамрита Прабху видит все три). **Внешние подотчётники — тоже custodial-счета:** Ашиш/₹. Выдал под отчёт = перевод на его счёт; отчитался чеками = расходы с его счёта; вернул = перевод обратно. Отдельного механизма подотчётов нет.

**Политика минуса:** реальный счёт не может уйти в минус для операций пользователя счетов и обычного расхода (RPC отклоняет); **исключение** — `reversal` и `reconciliation_adjustment`, выполняемые администратором, могут увести реальный счёт в минус с явным предупреждением на главном экране и требованием внеплановой сверки (chargeback и отмена онлайн-платежа физически создают такую ситуацию — запрет отражать реальность был бы хуже минуса). Подотчётный счёт — минус всегда предупреждение, не запрет.

### 4.2 fin_operations — операции

| Поле | Тип | Описание |
|------|-----|----------|
| id | uuid PK | Генерируется на клиенте при открытии формы |
| type | enum | `payment` / `refund` / `transfer` / `expense` / `income` / `donation` / `opening` / `reversal` / `reconciliation_adjustment` |
| occurred_on | date | Фактическая дата события (IST). Не ограничена чекпоинтом — ledger_seq уже защищает зафиксированный остаток (см. инвариант 5). **Для `reversal` — зависит от причины:** если это исправление ошибки ввода, реального движения денег не было → `occurred_on` = `occurred_on` исходной операции (`created_at` отдельно показывает реальную дату исправления); если это реальное обратное движение денег (chargeback, отмена онлайн-платежа банком) → `occurred_on` = фактическая дата этого обратного движения, не дата исходного платежа. Без этого разграничения месячные отчёты зависели бы от того, какую дату администратор случайно выбрал в форме |
| approval | enum | `pending` / `approved` / `disputed` / `not_required` |
| is_reversed | bool | Операция сторнирована целиком (информационный флаг; проводки остаются в остатках). Частичного сторно нет |
| is_post_close | bool | Сводный признак: true, если хотя бы одна проводка операции — post-close. **Не источник истины и не обязан быть хранимой колонкой** — предпочтительный вариант: вычисляемое поле или view (`bool_or(fin_postings.is_post_close)` по `operation_id`), не физическая колонка, которую пришлось бы поддерживать триггером при каждом изменении состава проводок. Если всё же хранится физически — обновляется только триггером/RPC, клиент это поле никогда не передаёт. Используется только для отображения в списках операций. Формулы, доступ и проверки всегда смотрят на признак конкретной проводки, не операции — одна операция может содержать проводки по разным объектам, один из которых уже закрыт, другой нет |
| original_operation_id | FK | Для `reversal` — исходная операция (обязательно) |
| payer_contact_id | FK → contacts | Плательщик для `payment`; жертвователь для именного `donation` (не являющегося платежом участника). Может отличаться от участников на проводках. UI подписывает поле по типу операции: «Плательщик» или «Жертвователь» |
| refund_recipient_contact_id | FK → contacts | Для `refund`: получатель денег, если отличается от плательщика исходного платежа. Nullable — по умолчанию получатель равен `payer_contact_id` исходной операции |
| reason | text | Обязательна для `reversal`, `reconciliation_adjustment`, операций с `is_post_close = true`, а также при простановке `approval = disputed` (руководитель департамента должен видеть, в чём именно спор) |
| request_hash | text | Вычисляется сервером в RPC по нормализованному содержимому формы (клиент передаёт только payload, не сам hash — иначе идемпотентность можно подделать). Идемпотентность: тот же `id` (client UUID) + тот же вычисленный hash → возвращается уже созданная операция; тот же `id`, другое содержимое → явный конфликт (ошибка, не молчаливый no-op) |
| created_by | FK → users | |
| created_at | timestamptz | |
| comment | text | |

Операции департаментов рождаются `pending`; операции администратора — `not_required`.

**Отклонение траты департамента больше не равно автоматическому сторно** (см. инвариант 4 и UC-10): администратор ставит `disputed`, если деньги физически не возвращены, и решает дальше по правилу раздела 8; сторно — только если операция была ошибкой ввода без движения денег.

### 4.3 fin_postings — проводки

Денежные строки операций. Аналитика — прямо здесь.

| Поле | Тип | Описание |
|------|-----|----------|
| id | uuid PK | Клиентский UUID |
| ledger_seq | bigserial | Монотонная последовательность вставки, глобальная по таблице. Присваивается **только внутри RPC после получения блокировки затронутых счетов** (`FOR UPDATE`), не раньше — иначе порядок seq мог бы разойтись с порядком коммитов транзакций. Не пересчитывается, не переиспользуется. Основа чекпоинта сверки (см. 4.9 и инвариант 5) |
| operation_id | FK | |
| account_id | FK → fin_accounts | |
| direction | enum | `in` / `out` |
| amount | numeric(14,2) | > 0 |
| currency | FK | = валюте счёта (constraint) |
| amount_base | numeric(14,2) | В ₹; фиксируется при проведении по действующему курсу; не пересчитывается |
| rate_used | numeric(12,6) | Применённый курс (1.0 для ₹) |
| category_id | FK → fin_categories | Статья. Обязательна для `payment`/`refund`/`expense`/`income`/`donation`; nullable для `transfer`/`opening`/`reconciliation_adjustment` (внутреннее движение, не входит в отчёт по статьям). Для `reversal` — наследуется от зеркалируемой проводки без проверки направления (см. 4.5) |
| cost_center_id | FK → fin_cost_centers | Подразделение-бенефициар. RPC записывает его при создании проводки: явно указанное значение (трата за другой департамент) или `account.default_cost_center_id` как значение по умолчанию — в обоих случаях это финальное, неизменное значение проводки, не ссылка для последующего пересчёта. Null для `transfer`/`opening` (раздел 7 — не входят в аналитику подразделений) |
| object_id | FK → fin_accounting_objects | Учётный объект; nullable = общие расходы ШРСК. В MVP объект всегда типа `retreat` — см. 4.13 |
| is_post_close | bool NOT NULL default false | **Источник истины** для post-close-статуса — свойство конкретной проводки относительно её `object_id`, не операции целиком. Значение устанавливает только сервер под блокировкой объекта (раздел 5, инвариант 6) — по тому, существует ли уже initial closure для `object_id` этой проводки в момент записи. Проводки без `object_id` всегда `false` |
| participant_id | FK → contacts | nullable |
| participant_balance_kind | enum | `none` / `org_fee` / `accommodation` / `meals` / `extra` / `general`. Обязательно, если `participant_id` задан. `none` — проводка касается участника (например, его именное пожертвование), но не входит в расчёт его долга/аванса. `org_fee`/`accommodation`/`meals`/`extra` — платёж закреплён за конкретным блоком (совпадает с `fin_charges.kind`). `general` — платёж не закреплён ни за одним блоком, распределяется алгоритмом раздела 7 по порядку org_fee → accommodation → meals → extra |
| refund_of_posting_id | FK → fin_postings | Только для `refund`: конкретная исходная платёжная проводка (не операция). Для MVP один `refund` = ровно одна проводка на одну исходную (раздел 10) — возврат нескольким участникам одного группового платежа оформляется несколькими отдельными операциями `refund`, а не одной операцией с несколькими проводками. `amount_base` возврата — пропорционально от исходной проводки, лимит — по невозвращённому остатку net (формулы — раздел 7) |
| contractor_id | FK → fin_contractors | nullable |
| payment_channel | enum | `cash` / `bank_transfer` / `card` / `paypal`; nullable. Валюта уже определяется счётом проводки — отдельного канала на валюту (вроде `online_rub`) не нужно, `bank_transfer` покрывает любой безналичный перевод в любой валюте счёта |

**Составные случаи:**
- Перевод = две проводки (out + in) одной операции; при разных валютах `amount_base` обеих равен (фактический курс обмена фиксируется соотношением сумм). Если ни одна из валют не ₹ (например, USD → EUR через обменник): `amount_base` исходящей ноги берётся по курсу USD→INR из `fin_exchange_rates`, `rate_used` входящей ноги (EUR→INR) выводится из равенства `amount_base` — так каждая нога всё равно имеет собственный корректный курс к ₹, хотя напрямую курс USD→EUR не хранится
- Мультивалютный платёж (часть ₹ наличными + часть $ PayPal) = несколько in-проводок одной операции
- Групповой платёж = несколько in-проводок по одному счёту, по одной на участника (participant_id на каждой)
- Расход «70% ретрит ЛК / 30% общие» = две проводки с суммами

### 4.6 fin_currencies и 4.7 fin_exchange_rates

Валюты: code (INR/RUB/USD/EUR), symbol, name, is_active. Все с двумя знаками после запятой.

Курсы: `object_id` (FK → `fin_accounting_objects`, nullable = общий курс — та же косвенность, что и у проводок, иначе фаза 2 снова потребует миграции схемы для гестхауса), effective_date, from_currency → INR, rate, created_by. `UNIQUE NULLS NOT DISTINCT (object_id, effective_date, from_currency)` — обычный Postgres `UNIQUE` не считает несколько NULL дублями, а для общего курса (`object_id = NULL`) дубли так же не нужны, как и для курса конкретного объекта. Правило выбора курса — детерминированный алгоритм (не «курс объекта, если он вообще есть», а с учётом даты):
```text
1. Искать курс с object_id = проводки: MAX(effective_date) WHERE effective_date ≤ occurred_on.
2. Не найден → искать общий курс (object_id IS NULL): MAX(effective_date) WHERE effective_date ≤ occurred_on.
3. Не найден ни один → отказ проведения (администратор сначала вносит курс).
```
Обратные и кросс-курсы вычисляются.

### 4.9 fin_reconciliations — сверки

| Поле | Тип | Описание |
|------|-----|----------|
| id | uuid PK | Клиентский `request_id` команды `perform_reconciliation` — не серверный генератор |
| request_hash | text NOT NULL | Серверный hash нормализованного payload: `account_id`, `opened_seq`, **серверно вычисленный** `counted_balance` для `cash_count`-счетов или переданный `statement_balance` для `statement`-счетов (не произвольное клиентское значение), нормализованные `counts`, `adjustment_reason`, `comment`. Нормализация `counts` перед хешированием — одно фиксированное правило, без вариантов: точки хранения сортируются по имени; номиналы внутри точки — по возрастанию; **дубль одной точки хранения в одном запросе — отказ** (не молчаливое объединение — это следует диагностировать как ошибку ввода, а не тихо чинить); **нулевые количества номинала удаляются** перед хешированием (не переносятся как есть); `other_amount`/`other_comment` участвуют в hash как есть; неизвестные поля JSON — отказ, не игнорирование. Без этого два логически одинаковых запроса с одним набором купюр, но разным порядком полей или мусорным клиентским значением остатка, дали бы разный hash и не считались бы повтором. Идемпотентность: тот же `id` + тот же `request_hash` → возвращается уже созданная сверка, ничего не создаётся заново; тот же `id` + другой `request_hash` → `idempotency_conflict`. Тот же механизм, что у операций (4.2) и у закрытий (4.15) — нужен и здесь, потому что у сбалансированной сверки (без adjustment) нет `adjustment_operation_id`, за который можно было бы зацепить идемпотентность |
| account_id | | Сверяемый счёт |
| performed_at, performed_by | | |
| system_balance | numeric | Расчётный остаток **по проводкам с `ledger_seq ≤ cutoff_ledger_seq`**, не по дате |
| cutoff_ledger_seq | bigint | Снэпшот максимального `ledger_seq` счёта на момент сверки. Определяет, какие проводки считаются подтверждёнными этим чекпоинтом |
| adjustment_operation_id | FK → fin_operations | Nullable. Если сверка закрыта корректировкой — ссылка на созданную `reconciliation_adjustment` |
| counted_balance | numeric | Фактический — сохранённый итог, режим определяется `fin_accounts.reconciliation_mode` того счёта (не полем этой таблицы, они всегда согласованы через `perform_reconciliation`): `cash_count` — **не принимается от клиента, а вычисляется сервером** как Σ(номинал × количество) по `counts`; клиент передаёт только `counts`, а не готовую сумму — иначе испорченный клиент мог бы прислать `counts`, соответствующие 100 000, но `counted_balance = 90 000`, и система зафиксировала бы несуществующую недостачу 10 000. `statement` — `counts` обязан быть `NULL`, значение приходит от клиента как `statement_balance` из выписки напрямую |
| difference | numeric | `counted_balance − system_balance` (signed: положительная — фактических денег больше расчётных, отрицательная — меньше). Подсвечивается |
| original_difference | numeric | Nullable. Если сверка закрыта через `reconciliation_adjustment` — `counted_balance − system_balance_before_adjustment` (тот же знак, что у `difference`, посчитанный до корректировки — после неё `difference` станет нулём). Знак определяет направление adjustment: положительная разница → adjustment `in`, отрицательная → adjustment `out` |
| counts | jsonb | Пересчёт наличных: `[{"location": "На руках", "denominations": {"500": 12, "200": 7, "100": 40}, "other_amount": 35, "other_comment": "мелочь и монеты"}, {"location": "Сейф", "denominations": {"2000": 10, "500": 30}}]`. `other_amount` — необязательная сумма монет/дробных остатков, не сводимых к перечисленным номиналам валюты счёта; при наличии — обязателен `other_comment`. Для безналичных — null (сверка с выпиской). RPC проверяет: количество каждого номинала — целое неотрицательное; номинал разрешён для валюты счёта (справочник, раздел 9.5); `reconciliation_mode = statement` ⇒ `counts` обязан быть `NULL` (иначе отказ — нельзя одновременно вводить и пересчёт купюр, и остаток по выписке); `reconciliation_mode = cash_count` ⇒ `counts` обязателен |
| is_checkpoint | bool | Сошлось или расхождение принято корректирующей операцией |
| comment | text | Сюда же — факт передачи ответственности за счёт, принятые расхождения |

Точки хранения — строки в jsonb (их 3–4: на руках, сейф, при необходимости другие; состав стабилен, отдельный справочник не нужен). Номиналы: ₹ 2000/500/200/100/50/20/10; ₽ 5000/2000/1000/500/200/100/50; $ 100/50/20/10/5/1; € аналогично.

**Боксы пожертвований — не точки хранения:** содержимое бокса неизвестно системе до выемки и в расчётный остаток кассы не входит. Выемка = пересчёт + приходная операция `donation`.

**Почему `ledger_seq`, а не дата:** новая проводка всегда получает `ledger_seq` больше любого существующего на момент вставки — независимо от того, какую `occurred_on` ей проставили. Поэтому проводка, внесённая после чекпоинта, никогда не попадёт в уже зафиксированный `system_balance` этого чекпоинта.

**Защита от гонки при сохранении:** форма сверки при открытии запоминает `opened_seq = MAX(ledger_seq)` счёта и показывает остаток на этот момент. При сохранении RPC под блокировкой счёта (`FOR UPDATE`) проверяет текущий `MAX(ledger_seq)`: если он больше `opened_seq` (кто-то успел провести операцию, пока форма была открыта) — сохранение отклоняется с `reconciliation_stale`, администратор обновляет форму и пересчитывает; иначе чекпоинт сохраняется ровно на `opened_seq`. Без этой проверки поздняя операция, вставленная между открытием формы и её сохранением, могла бы попасть в `cutoff_ledger_seq` чекпоинта, хотя администратор её не видел и не считал.

### 4.10 fin_audit_log

Один триггер на все fin_-таблицы: id, entity, entity_id, action (`insert`/`update`/`delete`), before, after (jsonb), user_id, at, **reason** (nullable text), **request_id** (nullable uuid), **correlation_id** (nullable uuid). Append-only — приложение не может редактировать и удалять строки лога.

Для state-команд (`update_posting_analytics`, `set_approval`, `transfer_account_responsibility`, `cancel_charge` и т.п.) причина обязана попасть в лог, хотя сама команда не создаёт собственной сущности с полем `reason` — иначе можно узнать, **что** изменилось, но не **почему**. Решение без новой таблицы: каждая изменяющая RPC перед DML устанавливает контекст транзакции (`set_config('app.change_reason', ..., true)`, `set_config('app.request_id', ..., true)`), аудит-триггер копирует эти значения в `reason`/`request_id` наравне с обычными `before`/`after`.

### 4.11 fin_account_access

user_id + account_id. Пользователь видит и оперирует счетами из своего набора. Администратор и наблюдатель — все счета без записей здесь.

**Явная граница приватности** (implementation guide должен превратить это в конкретные views/RLS-политики, не оставлять на усмотрение реализации): доступ к счёту НЕ означает доступ к посторонней информации внутри операций на этом счёте — пользователь счетов видит свои проводки, но не других участников группового платежа, не плательщика чужого платежа, не проводки по чужим счетам той же операции, не вложения, относящиеся к чужому счёту.

**Upload-flow** (implementation guide должен зафиксировать один процесс, не оставлять на усмотрение реализации — Postgres-транзакция и загрузка файла в Storage не атомарны друг относительно друга): 1) клиент загружает файл во временный приватный path; 2) получает временный token/path; 3) создаёт операцию (`create_expense` и т.п.) — pending-операция может временно существовать без чека; 4) следующий вызов привязывает уже загруженный файл к реально созданным `operation_id`/`posting_id` (создаёт строку `fin_attachments`); 5) незакреплённые временные файлы старше суток удаляются фоновой задачей.

Без этого шага opening-коррекция закрытого ретрита молча разошлась бы с уже выпущенным PDF: цифры в системе изменились, а UI продолжал бы считать последний отчёт актуальным. Идемпотентна по (`id`=request UUID, серверный `request_hash`) — тот же запрос дважды не создаёт вторую компенсирующую строку.

### 4.15 fin_object_closures — версии финансового закрытия

Финансовое закрытие учётного объекта (ретрита) — не boolean-флаг, а версионируемая запись: у объекта может быть несколько закрытий (initial + переиздания после post-close операций).

| Поле | Тип | Описание |
|------|-----|----------|

Тот же механизм, что у `fin_operations` (раздел 4.2), применённый к закрытию — иначе двойной клик создаёт дубль-версию, а повтор после сетевой ошибки возвращает ошибку вместо уже созданного результата |

## 5. Инварианты (7)

1. **Один счёт — одна валюта.** Валюта проводки = валюте счёта. Мультивалютность точки — через несколько счетов с общей group_name

2. **Денежные поля проведённой операции неизменны:** счёт, сумма, валюта, направление, дата, `amount_base`, `rate_used`. Перенос проводки на другой объект учёта (правка `object_id`) не пересчитывает `amount_base` — если курс был внесён неверно, это ошибка ввода, исправляемая только `reversal` + новый ввод по правильному курсу, а не правкой `object_id`. Исправление денег — только сторно (`reversal` со ссылкой на исходную и причиной; зеркальные проводки, всегда полные — частичного сторно не существует). **Аналитические поля** (статья, cost center, объект учёта, участник, `participant_balance_kind`, контрагент, комментарий) правятся администратором с аудит-следом. Правило зависит от того, когда проводка проведена относительно initial closure её объекта — не единое «до закрытия», а два разных случая: **проводки, проведённые до initial closure — аналитика заморожена навсегда** после закрытия (никаких исключений, это и есть цена фиксации отчёта); **post-close проводки (созданные после initial closure, `is_post_close = true`) — аналитику можно править администратору и после их создания**, под блокировкой объекта, с причиной, и это, как любая другая post-close-правка, ставит `report_dirty_at` (раздел 5) — не нужно сторнировать реальное денежное движение только ради опечатки в статье. Старые версии PDF при этом не меняются, текущий отчёт просто снова становится dirty. **Перенос post-close-проводки между объектами** (например, поздний расход ошибочно отнесён к закрытому A, переносится в B) выполняется одной RPC по алгоритму: 1) заблокировать старый и новый объект (оба, в порядке UUID); 2) старому объекту, если он закрыт, проставить `report_dirty_at`; 3) изменить `object_id`; 4) пересчитать `is_post_close` этой проводки заново — `true`, если у нового объекта уже есть initial closure, `false`, если новый объект открыт или `object_id = NULL`; 5) если новый объект тоже закрыт — проставить `report_dirty_at` и ему; 6) сохранить обязательную причину в аудит-лог. `is_post_close` не «наследуется» и не остаётся прежним при переносе — это вычисляемое на момент записи свойство пары «проводка × её текущий объект», а не история проводки. **Исключение при активном `refund` (без reversal):** поля `participant_id`, `participant_balance_kind`, `object_id` и `payer_contact_id` редактированию не подлежат — иначе платёж и его возврат разъедутся по разным участникам/объектам, а сальдо станет неверным на обеих сторонах. **Исключение при активном `reversal` (операция уже полностью сторнирована, `is_reversed = true`): все аналитические поля исходных и зеркальных проводок неизменяемы независимо от post-close-статуса —** `category_id`, `cost_center_id`, `object_id`, `participant_id`, `participant_balance_kind`, `contractor_id`; `payer_contact_id` исходной операции — тоже. Не «дополнительно к списку для refund», а полный самостоятельный список — если читать только «дополнительно», легко ошибочно вывести, что заморожены лишь три новых поля, а не все семь. Reversal наследует эти поля от источника **в момент создания сторно**; если после этого поменять их у исходной (или у reversal) проводки, взаимопогашение по статье/подразделению перестаёт быть нулевым — деньги в сумме сойдутся, но отчёт по статьям и cost center станет ложным (пример: расход «Электричество/Кафе» сторнирован, затем исходную проводку перевели в «Зарплата/Кухня» — сумма 0, но отчёты показывают несуществующий +100 000 по «Электричество/Кафе» и −100 000 по «Зарплата/Кухня»). Комментарий можно дополнять — он не входит в список замороженных полей. Правку в этом случае делает только повторный ввод новой операции с правильными атрибутами, исходная остаётся как есть; такую правку (перенос при наличии зависимостей) выполняет только полный `reversal` исходной операции (что само по себе запрещено при ненулевом `net_refunded`, инвариант 6) с последующим повторным вводом — то есть перенос делается через отмену и новый ввод, а не прямую правку. **`refund_recipient_contact_id`** после проведения `refund` неизменяем вообще — ошибочный получатель исправляется `reversal` этого refund (разрешён, если по нему самому нет вложенных зависимостей) и повторным `refund` с верным получателем. При создании `refund` сервер всегда сам подставляет `payer_contact_id` исходного платежа как получателя по умолчанию — клиентский payload на это поле не влияет, если явно не передан `refund_recipient_contact_id` с причиной (UC-7)

3. **Остатки нигде не хранятся** — вычисляются из проведённых проводок. Сторнированные операции остаются в расчёте, их гасят зеркальные проводки reversal

4. **Согласование не влияет на деньги.** Pending-операция входит в остатки сразу (деньги физически потрачены); в аналитике — с пометкой «не проверено». Администратор ставит `approved`, `disputed` или инициирует `reversal` — но не автоматически. `disputed` не двигает деньги: используется, когда чек не принят или назначение спорно, а деньги физически не возвращены. `reversal` уместен только если операция была ошибкой ввода без реального движения денег. Если деньги реально нужно вернуть — отдельная операция (перевод, новая трата), а не сторно исходной

5. **Чекпоинт сверки фиксирует остаток по `ledger_seq`, не по дате.** `system_balance` в `fin_reconciliations` вычисляется по проводкам с `ledger_seq ≤ cutoff_ledger_seq`. `occurred_on` — только фактическая дата события для аналитики; чекпоинт от неё не зависит, поэтому дата операции **не ограничена** датой последнего чекпоинта. Операция может быть внесена в систему позже своей фактической даты (`created_at` ≠ `occurred_on`) — в ДДС такая проводка помечается «внесено позднее», но попадает в аналитический период своей фактической даты, а не искажает его задним числом внесённой сегодняшним числом операцией. Форма сверки защищена от гонки отдельно: см. 4.9. **Вся сверка — один атомарный RPC `perform_reconciliation(account_id, opened_seq, statement_balance nullable, counts nullable, adjustment_reason nullable, request_id)`** — не общий параметр «counted_balance», а два раздельных, ровно один из которых заполнен согласно `fin_accounts.reconciliation_mode` счёта (4.1): `statement_balance` для `statement`-счетов, `counts` для `cash_count` (сервер сам считает по ним `counted_balance`). Не последовательность из независимых вызовов: под блокировкой счёта RPC (1) проверяет `current_seq = opened_seq` (иначе `reconciliation_stale`), (2) вычисляет `original_difference`, (3) при ненулевой разнице создаёт `reconciliation_adjustment` внутренним вызовом (не отдельной командой — см. раздел 10), (4) получает для неё `ledger_seq`, (5) пересчитывает `system_balance`, (6) создаёт строку `fin_reconciliations` с `original_difference` и `adjustment_operation_id`, (7) сохраняет `cutoff_ledger_seq`. Промежуточных состояний «корректировка создана, чекпоинт ещё нет» не бывает — либо вся транзакция целиком, либо ничего

6. **Составные операции атомарны** — перевод, платёж, сторно, корректировка сверки создаются одной RPC (Postgres-функцией) целиком или никак. Id — клиентские UUID; идемпотентность — по паре (UUID, hash содержимого, вычисленным сервером): повтор той же формы безвреден и возвращает созданную операцию, повтор с другим содержимым — явная ошибка конфликта.

**Идемпотентность проверяется первым шагом, раньше любых предусловий, которые сама операция могла изменить.** Это касается **двух разных контрактов**, и они не взаимозаменяемы:

**Command-идемпотентность (по клиентскому UUID + серверному `request_hash`)** — для RPC, создающих самостоятельную новую сущность со своей строкой: `create_payment`, `create_refund`, `create_reversal`, `create_transfer`, `create_expense`, `create_income`, `create_donation`, `create_opening`, `create_charge`, `perform_reconciliation`, `create_closure`, `reissue_closure`, `create_opening_correction`, `create_attachment` (`id` — клиентский UUID; `storage_path` — `UNIQUE`, повторная привязка того же временного файла не создаёт вторую metadata-запись). Единый порядок:
1. Сервер канонизирует payload и вычисляет `request_hash`;
2. ищет существующую запись по клиентскому UUID;
3. нашёл + hash совпадает → сразу возвращает прежний результат, дальше не идёт;
4. нашёл + hash отличается → `idempotency_conflict`;
5. не нашёл → только теперь блокировки и бизнес-проверки (`reconciliation_stale`, `report_not_dirty` и т.п.).
Если проверять `reconciliation_stale`/`report_not_dirty` раньше идемпотентности, повтор запроса после сетевого сбоя не вернёт прежний результат: первый (успешный, но не увиденный клиентом) вызов уже сам сдвинул `ledger_seq` или обнулил `report_dirty_at`, и retry упадёт на предусловии вместо того, чтобы найтись по idempotency key.

**State-идемпотентность (без отдельного request UUID)** — для RPC, которые лишь переводят уже существующую сущность в целевое состояние, а не создают новую строку: `cancel_charge`, `set_approval`, `update_posting_analytics`, `finalize_closure`, `transfer_account_responsibility`. Здесь нет отдельной таблицы для хранения запроса — идемпотентность определяется сравнением текущего состояния с запрошенным:
```text
cancel_charge:      уже is_cancelled=true с той же cancelled_reason → вернуть текущий результат (no-op);
                     уже отменено с другой причиной → conflict
set_approval:        операция уже в целевом статусе с теми же переданными данными → success/no-op;
                     в другом статусе или с другой причиной → обычная бизнес-проверка перехода
update_posting_analytics: текущие аналитические поля уже равны целевым → success/no-op;
                     иначе — правка под блокировкой, если разрешена инвариантами. Клиент передаёт
                     expected_audit_version (или hash текущего набора аналитических полей) — RPC
                     отклоняет правку, если проводку кто-то уже успел изменить между чтением
                     формы и отправкой (защита от потерянных обновлений, не только от повтора)
finalize_closure:    уже finalized с тем же attachment_id → вернуть текущий результат;
                     finalized с другим attachment_id → conflict
transfer_account_responsibility: responsible_person уже равен new_responsible_person И
                     текущий MAX(ledger_seq) счёта равен ожидаемому на момент вызова →
                     success/no-op; responsible_person уже изменён на кого-то другого →
                     responsibility_conflict. Отдельного `request_id`/hash не нужно — как и
                     остальным state-командам, ей достаточно сравнения текущего состояния с целевым
```
Это не ослабление контракта, а честное признание: не для каждого RPC применима одна и та же схема «request UUID + hash» — командам создания нужна первая, командам перехода состояния — вторая.

**Единый порядок блокировок во всех RPC** (нарушение порядка — источник deadlock, поэтому порядок один для всех функций):
1. Учётные объекты (`fin_accounting_objects`), если операция их затрагивает — по UUID, при нескольких объектах в отсортированном порядке;
2. Исходная операция и её проводки, если RPC зависит от чего-то уже существующего (`refund`, `reversal`, правка аналитики, привязка approval) — по UUID;
3. Денежные счета (`fin_accounts`) — по `account_id` (`ORDER BY account_id` при двух и более, иначе два встречных перевода дают deadlock);
4. Проверки (остаток, лимит возврата, наличие closure) и выделение `ledger_seq`;
5. Запись.

**Когда набор объектов для шага 1 известен только из исходной проводки (refund, reversal, перенос `object_id`) — нужен протокол «прочитать → заблокировать → перепроверить»**, а не блокировка вслепую по предварительно прочитанному значению:
```text
1. Прочитать object_id исходной проводки БЕЗ блокировки (предварительное значение).
2. Заблокировать найденные объекты в порядке UUID.
3. Заблокировать исходную операцию и её проводки.
4. Повторно прочитать object_id ПОД блокировкой.
5. Если значение изменилось (кто-то успел перенести проводку между шагом 1 и 3) —
   заблокированный набор объектов неверен: откатить и повторить с начала с новым значением.
6. Только теперь блокировать денежные счета (шаг 3 общего порядка) и продолжать.
```
Без этого шага `refund` мог бы заблокировать старый объект A, пока проводка уже переехала в объект B, и оставить закрытие B несогласованным с этим refund.

Проверка остатка перед списанием — с блокировкой строк счетов (`SELECT ... FOR UPDATE`), захватываемой **до** выделения `ledger_seq`. Расход сверх остатка: реальный счёт пользователя счетов — отказ; подотчётный — предупреждение; `reversal` и `reconciliation_adjustment` администратора — минус разрешён с предупреждением (это отражение реальных chargeback/отмен, а не обычная трата).

**Полное сторно платежа запрещено, если по нему уже есть хотя бы один активный (не сторнированный) `refund`** — то есть `net_refunded = Σ refund.amount − Σ reversal(refund).amount > 0`. При `net_refunded = 0` полное сторно разрешено как обычно; при `net_refunded = original_amount` — сторно и дополнительный refund не имеют смысла, RPC отклонит оба; в промежуточных случаях доступен только дополнительный `refund` на `original_amount − net_refunded` (раздел 7, раздел 8).

**Исходная проводка — тоже объект блокировки (шаг 2 выше), не только счета и объекты.** `create_refund` сначала блокирует строку `fin_postings` по `refund_of_posting_id` (`FOR UPDATE`), пересчитывает `net_refunded` под этой блокировкой и только затем блокирует расходный счёт и пишет — иначе два одновременных `refund` по 70 от платежа 100 могут оба прочитать остаток 100 до того, как другой закоммитится, и вместе вернуть 140. По той же причине `create_reversal(payment)` блокирует исходную операцию и все её проводки, а RPC правки аналитической проводки (`update_posting_analytics` — правка `participant_id`/`participant_balance_kind`/`object_id`/`category_id`/`cost_center_id`) блокирует саму проводку **перед** проверкой отсутствия зависимых refund/reversal (инвариант 2) — иначе правка и создание refund той же проводки могут разминуться.

**Блокировка объекта учёта (шаг 1) требуется для любого RPC, который может изменить состав или содержимое `totals_snapshot` закрытого или закрываемого объекта**, а не только для создания проводки или начисления: создание проводки с `object_id`; правка любого аналитического поля уже существующей проводки, привязанной к объекту (`category_id`, `cost_center_id`, `object_id`, `participant_id`, `participant_balance_kind`); перевод `approval` операции, привязанной к объекту, в `disputed` или из него; создание, отмена или замена начисления по объекту; **создание opening-коррекции (`create_opening_correction`) по участнику этого объекта** — она меняет его долг/аванс так же, как начисление или платёж, значит и snapshot; создание initial или очередной версии `fin_object_closures`. При переносе проводки между объектами (A → B) блокируются оба в порядке UUID — тогда `create_closure` объекта A либо застаёт проводку ещё в A и включает её в snapshot целиком, либо не застаёт вовсе (перенос уже закоммичен раньше) — состояния «наполовину» не бывает. `create_closure` под той же блокировкой повторно проверяет отсутствие pending и **disputed**-операций непосредственно перед фиксацией snapshot.

7. **Всё пишется в аудит-лог:** правка аналитики, сторно, согласование, справочники, курсы, доступ. Видно: что, кто, когда, было/стало. Триггер стоит на всех `fin_`-таблицах, кроме самой `fin_audit_log` (не аудитирует себя). Наблюдателю лог отдаётся через view с маскированием персональных данных, путей вложений, причин скидок и служебных полей доступа

**Правило закрытия учётного объекта (ретрита):** финансовое закрытие — не флаг, а запись в `fin_object_closures` (4.15). Администратор инициирует закрытие, когда учёт по объекту завершён (долги отработаны или осознанно зафиксированы, **нет pending и нет нерешённых disputed-операций**): RPC под блокировкой объекта пересчитывает `totals_snapshot`, создаёт `fin_object_closures` с `version = 1`, `is_initial = true`, `status = report_pending`, `closed_at/closed_by` — объект уже закрыт для новых операций на этом шаге, деньги ни от чего не зависят. PDF формируется и загружается отдельно (Edge Function из сохранённого `totals_snapshot`), после чего `finalize_closure` связывает уже загруженный файл — проставляет `attachment_id`, `finalized_at/finalized_by`, `status = finalized`; сбой генерации не откатывает закрытие и не блокирует денежные операции. После закрытия:

- Новые операции и правка аналитики по объекту — запрещены **как общее правило**;
- **Исключение (post-close), только роль Администратор:** `payment`, `refund`, `income`, `expense`, `donation`, `reversal`, `reconciliation_adjustment` по объекту разрешены администратору с обязательной причиной и пометкой `is_post_close = true` — жизнь после закрытия продолжается (поздний платёж, возврат, chargeback, обнаруженная комиссия, поздний документ департамента). **Пользователь счетов создать такую операцию не может ни при каких условиях** — если хотя бы одна проводка создаваемой операции получила бы `is_post_close = true`, RPC требует роль администратора для всей команды целиком (операцию с проводками по нескольким объектам, из которых один закрыт, нельзя безопасно разделить по правам внутри одного вызова); пользователь счетов при попытке провести операцию по закрытому объекту получает отказ `object_closed`, а не автоматически становится post-close-автором. Правка аналитических полей уже проведённых до закрытия проводок по-прежнему запрещена;
- Создание, отмена или замена начисления (`fin_charges`) после закрытия — только администратором, с причиной (`creation_reason`/`cancelled_reason`);
- **`approval` операции, привязанной к объекту:** до initial closure — меняется под блокировкой объекта как обычно (включая переходы в/из `disputed`, инвариант 6). После initial closure — `approval` уже проведённых до закрытия операций не меняется вообще (ни в `disputed`, ни обратно): застарелый спор, обнаруженный после закрытия, — это не смена approval задним числом, а отдельное post-close административное действие с причиной и `report_dirty_at`, если он всё-таки требует денежной коррекции. Post-close операции всегда создаются сразу `not_required` — цикл согласования к ним не применяется;
- Любое из перечисленного выше, под той же блокировкой объекта, проставляет `fin_accounting_objects.report_dirty_at = now()` — явное поле, не вычисляемое сравнением таймстампов между разными таблицами (иначе отмену начисления, у которого нет собственной связанной `fin_operations`-записи, нечем было бы датировать относительно последней версии закрытия). UI показывает «Отчёт требует перевыпуска», пока `report_dirty_at` не null. Администратор нажимает «Перевыпустить PDF», когда удобно — `reissue_closure` под той же блокировкой объекта сначала проверяет `report_dirty_at IS NOT NULL` (иначе `report_not_dirty` — версия не создаётся; это защита от двух одновременных «Перевыпустить», из которых первый уже обнулил флаг: второй, получив блокировку позже, не создаёт лишнюю версию с теми же цифрами), затем пересчитывает `totals_snapshot`, создаёт следующую версию (`version + 1`, `is_initial = false`, обязательная `reason`) и обнуляет `report_dirty_at`; предыдущие версии не удаляются.

Практичная модель принята (альтернатива — «жёсткая»: закрывать только при нулевых долгах/авансах и гарантии отсутствия будущих операций — рассматривалась и отклонена как менее реалистичная для живого процесса).

**`is_post_close` — свойство проводки относительно её объекта, не операции целиком:** хранится в `fin_postings.is_post_close` (4.3), устанавливается сервером под блокировкой объекта при записи. Если операция затрагивает несколько объектов (редкий случай — например, разбивка расхода между двумя ретритами) и только один из них уже закрыт, `true` получают только проводки того конкретного (закрытого) объекта — `fin_operations.is_post_close` в этом случае лишь сводно отражает «хотя бы одна проводка post-close», ничего не решает и нигде не проверяется как условие. Начисления отдельного флага не получают — их post-close-статус виден через сравнение `fin_charges.created_at`/`cancelled_at` с моментом `is_initial`-версии закрытия объекта.

## 6. Use Cases

### Администратор

#### UC-1: Ежедневная сверка остатков

**Цель:** убедиться, что расчётный остаток каждого счёта совпадает с фактическим. Полная замена сверочных табличек Excel.

1. Администратор открывает «Сверка», выбирает счёт → система запоминает `opened_seq = MAX(ledger_seq)` счёта на этот момент
2. **Безналичный** (карта, PayPal): открывает выписку, сравнивает с расчётным остатком на экране
3. **Наличный:** для каждой точки хранения (на руках, сейф) — форма пересчёта по номиналам: ₹ 2000×__, 500×__, 200×__, 100×__, 50×__, 20×__, 10×__; автоподсчёт по точке и суммарно. Форма сохранена по запросу Ванамали Гопала — инструмент проверен его практикой
4. Итог сравнивается с расчётным остатком на `opened_seq`. Расхождение подсвечивается красным
5. **При расхождении:** система подсказывает номер и дату последнего чекпоинта («сверено по проводку №1842, 15.07 — проверьте операции после этого номера») → администратор ищет ошибку в операциях периода → находит: сторно + правильный ввод, пересчёт повторяется; не находит — переходит к шагу 6 с указанием причины расхождения
6. «Сохранить сверку» — одна кнопка, один вызов `perform_reconciliation` (раздел 5, инвариант 5), а не последовательность из создания корректировки и отдельного сохранения: под блокировкой счёта RPC проверяет `current_seq = opened_seq` (иначе `reconciliation_stale` — форма обновляется, пересчёт повторяется); если разница не найдена (шаг 5) и подтверждена причина — тем же вызовом создаёт `reconciliation_adjustment`, фиксирует `original_difference` и `adjustment_operation_id`; в любом случае сохраняет `cutoff_ledger_seq = opened_seq` (или новый seq, включающий только что созданную корректировку). Не бывает состояния «корректировка уже создана, чекпоинт ещё нет»
7. Здесь же — беглый просмотр непроверенных операций дня по счёту (массовые чекбоксы в ДДС — привычный ритуал самоконтроля)

**Краевые случаи:** пересчёт прерван (свет отключили) — форма сохраняет введённое локально до отправки; сверка одного счёта не блокирует операции по другим; операция, внесённая после открытия формы сверки, либо не попадёт в чекпоинт (если её провели уже после сохранения), либо вызовет `reconciliation_stale` и потребует пересчёта (если успела попасть между открытием формы и сохранением) — попасть в чекпоинт незамеченной она не может.

#### UC-2: Управление счетами и доступом

1. Создание счёта: имя, тип (реальный/подотчётный), валюта, группа, ответственный
2. Начальный остаток — операция `opening` (только при инициализации, с комментарием). `category_id`, `object_id`, `cost_center_id`, `participant_id` у неё всегда `NULL` — это не доход периода, а стартовая точка отсчёта; в ДДС показывается отдельной строкой «Остаток на начало», не смешивается с приходами при подсчёте итогов за период. `create_opening` **всегда** (в том числе во время cutover) блокирует счёт и проверяет: на счёте нет активной (не сторнированной) `opening`-операции **и** нет вообще никаких проведённых операций — не «или cutover-режим»: во время cutover счета и так пусты после reset (раздел 11), поэтому отдельное исключение для cutover не нужно и только создало бы лазейку — при нём два разных загрузчика (или повторный запуск с новым UUID) могли бы создать по счёту два opening подряд, а идемпотентность по UUID/hash от этого не защищает (разные UUID — разные запросы). Исправление ошибочного денежного `opening`: **до первого чекпоинта счёта** — полный `reversal` ошибочного `opening` + новый replacement `opening` (RPC проверяет: исходный `opening` сторнирован, чекпоинта у счёта ещё нет, кроме этой пары `opening`/`reversal` на счёте больше ничего нет, после операции на счёте останется ровно один активный `opening`); **после первого чекпоинта** — только `perform_reconciliation` с `reconciliation_adjustment`, второй `opening` создавать нельзя ни при каких условиях
3. Назначение доступа: выбрать пользователя → отметить его счета
4. Деактивация — при нулевом остатке (иначе отказ с подсказкой «переведите остаток»)
5. Смена ответственного — одна RPC `transfer_account_responsibility(account_id, expected_seq, new_responsible_person)`, не два отдельных действия «сверка, затем правка `responsible_person`»: между ними могла бы появиться новая проводка, и передача прошла бы по уже устаревшему остатку. RPC блокирует счёт и проверяет **оба** условия: `latest_checkpoint.cutoff_ledger_seq = expected_seq` **И** `MAX(fin_postings.ledger_seq) по счёту = expected_seq` — одного совпадения чекпоинта недостаточно, потому что новая проводка после сверки не меняет last checkpoint, но меняет текущий максимум seq (пример: чекпоинт зафиксирован на seq=100, приходит расход с seq=101 — проверка «100=100» прошла бы, хотя после сверки уже было движение денег). Если хоть одно условие не совпадает — отказ `account_changed_since_reconciliation`, администратор сверяет счёт заново; только тогда меняет `responsible_person`. Факт передачи — в комментарии этой операции, не сверки. State-idempotent (раздел 5) — отдельного `request_id`/hash не требуется


#### UC-3: Перевод между счетами (включая обмен валют и выдачу под отчёт)

1. Счёт-источник → счёт-получатель, суммы
2. Одна валюта: сумма одна. Разные валюты: две суммы (отдал 83 500 ₹ — получил 1 000 $), курс фиксируется соотношением, `amount_base` обеих проводок равен
3. RPC создаёт операцию `transfer` с двумя проводками атомарно (с блокировкой обоих счетов)

**Примеры:** обмен валют Касса/₹ → Касса/$ · выдача департаменту Касса/₹ → Кафе/₹ (200 000) · выдача под отчёт Касса/₹ → Ашиш/₹ (10 000) · возврат Ашиш/₹ → Касса/₹ (2 000).

Перемещение наличных между точками хранения одной кассы (на руках → сейф) — **не операция**; отражается только раскладкой при сверке.

#### UC-4: Начисления участнику

1. Выбор участника и ретрита → «Добавить начисления»
2. Оргвзнос: выбор статуса (Полный / Детский / Волонтёр / …) → сумма из тарифа ретрита в ₹, копируется
3. Проживание: гостиница (Гест-хаус / здание ШРСК / внешняя) × тип номера (1/2/4-местный, выкуп) × ночей → цена в ₹ копируется из справочника гостиниц на момент создания
4. Питание: тариф × дней
5. Скидка / договорная цена: discount_amount + причина
6. Начисления появляются в карточке участника и в Guest Portal

**Краевые случаи:** участник переехал в другой номер — старое начисление проживания отменяется (is_cancelled + причина), создаётся новое по фактическим ночам; нестандартное размещение (двое в 4-местном) — договорная цена с причиной (правила уточняются с АК, открытый вопрос).

#### UC-5: Приём платежа от участника

1. Выбор участника и ретрита (включая будущие — предоплата)
2. Карточка: начислено (по блокам: оргвзнос / проживание / питание / дополнительно) / оплачено / **долг или аванс** по каждому блоку
3. Форма платежа — 5 блоков (оргвзнос / проживание / питание / дополнительно / пожертвование), в каждом: начислено-оплачено-остаток; строки сумм с валютой и каналом (наличные ₹ / наличные $ / онлайн ₽ / PayPal), кнопка «+» для второй валюты; конвертация в ₹ по курсу ретрита показывается сразу; «Сохранить всё» — одна операция. Блок «Дополнительно» видим, если по участнику есть хотя бы одно из: начисление `kind=extra`; opening-позиция `balance_kind=extra`; проводка (платёж/возврат) с `participant_balance_kind=extra`; ненулевой остаток этого блока — а не только наличие начисления, иначе opening-долг по `extra` может существовать при скрытой колонке
4. RPC: операция `payment` с in-проводками по соответствующим счетам (Касса/₹, Касса/$, PayPal/$…); каждая проводка получает `participant_balance_kind` по блоку формы, из которого она заполнена; строка «пожертвование» → `participant_balance_kind = none`
5. Долг/аванс по каждому блоку пересчитывается автоматически (формула раздела 7); пожертвование в этот расчёт не входит

**Предоплата:** ретрит будущий, начислений ещё нет → платёж просто проводится с `participant_balance_kind = general`; в карточке — аванс. Появились начисления — долг/аванс пересчитались сами, `general` распределяется по блокам в порядке оргвзнос → проживание → питание → дополнительно.

**Комиссия PayPal (правило):** две **отдельные операции**, не одна с двумя проводками — иначе сторно платежа автоматически «вернуло» бы и комиссию, которую PayPal обычно не возвращает: (1) `payment` на полную сумму gross (сколько заплатил участник — именно она зачитывается участнику), (2) отдельная `expense` на сумму комиссии со статьёй «Комиссия процессинга». Если процессор реально вернул комиссию — отдельно сторнируется операция (2).

#### UC-6: Групповой платёж

1. Плательщик (кто принёс деньги) + несколько участников
2. Одна валюта на групповой платёж (в двух валютах — два платежа, правило раздела 8)
3. Система предлагает распределение по долгам участников; администратор правит при необходимости
4. Одна операция, in-проводки по одному счёту — по одной на участника, каждая со своим `participant_balance_kind`

#### UC-7: Возврат участнику

1. Выбор исходной **проводки** (не операции — если платёж был групповым или мультивалютным, выбирается конкретная строка) → «Возврат»
2. Сумма ≤ невозвращённой части этой проводки (`original_amount − net_refunded`); по умолчанию — с того же счёта, куда пришёл платёж, получатель — плательщик исходной операции. Другой счёт допустим только в той же валюте и с причиной; получатель, отличный от плательщика, — тоже с причиной (`refund_recipient_contact_id`, желательно со ссылкой на документ)
3. Операция `refund` с out-проводкой (статья «Возврат участнику»), `refund_of_posting_id` → исходная проводка, `participant_balance_kind` наследуется от неё
4. Долг/аванс по соответствующему блоку пересчитались автоматически

#### UC-8: Пожертвования

- **Именное:** жертвователь из CONTACTS → `payer_contact_id` (то же поле, что у плательщика платежа — не отдельная колонка), сумма, валюта, канал, ретрит или без; если жертвователь одновременно является участником ретрита, проводка получает ещё и его `participant_id`, но `participant_balance_kind = none` (не гасит долг). Если жертвователь не участник — `participant_id` пуст, контакт виден только через `payer_contact_id`
- **Прасад-пожертвование:** отдельная статья прихода; в аналитике ретрита уменьшает чистый расход по статье «Прасад»
- **Выемка бокса, сумма полностью неизвестна заранее:** пересчёт содержимого → приход `donation` на пересчитанную сумму (источник в комментарии: «Бокс у входа, выемка 15.07»). До выемки содержимое бокса — вне учёта
- **Выемка бокса с известным изъятием («взял, верну»):** если сумма, которую кто-то взял из бокса, достоверно известна — это **не одна операция**, а две обычные: `donation` на полную пересчитанную сумму (например 15 000) + `transfer` касса → custodial-счёт человека на изъятую сумму (например 10 000). Если сумма изъятия неизвестна — в учёт входит только реально пересчитанное содержимое; нельзя одновременно утверждать неизвестный gross и учитывать изъятие

#### UC-9: Прямая трата или приход

Расход без подотчётника: электричество, зарплата, разовая оплата поставщику (contractor_id) → операция `expense`. Зеркально — обычный приход, не являющийся платежом участника или пожертвованием: возврат от контрагента, маржа Money Exchange, личное возмещение оспоренного расхода → операция `income`. Счёт → статья → сумма → объект учёта (ретрит) или общие → комментарий. Разбивка на несколько ретритов — несколькими проводками с суммами.

#### UC-10: Согласование операций департаментов

1. Главный экран: счётчик «Входящие» → список pending-операций: кто, счёт, сумма, статья, дата, комментарий, вложения-чеки
2. Администратор открывает операцию, смотрит чек
3. **Подтвердить:** при необходимости уточняет статью (переклассификация «Прочего»), проставляет объект учёта (система подсказывает по дате: попадает в период ретрита ± 3 дня) → `approved`
4. **Оспорить (`disputed`):** чек не принят, назначение спорно, деньги физически не возвращены → операция помечается `disputed`, деньги остаются в остатках (они реально потрачены), руководитель департамента видит статус и комментарий в своей ленте. Сторно **не создаётся** — деньги никуда не «возвращаются» фиктивно. Дальше решает человек по правилу раздела 8: ошибка ввода без движения денег → сторно; расход относится на другое подразделение → правка cost center; сумма становится задолженностью ответственного → остаётся на его custodial-счёте; деньги нужно вернуть организации — **два разных случая**: если деньги физически ещё у ответственного (просто не потрачены как надо) — обычный `transfer` с его custodial-счёта на кассу; если деньги действительно потрачены безвозвратно и ответственный возмещает **из личных средств** — `income` «Возмещение оспоренного расхода» со ссылкой на disputed-операцию в комментарии (перевод здесь не годится: он просто перемещает уже принадлежащие организации деньги внутри её счетов и не компенсирует реальную потерю)
5. Всё — с аудит-следом

#### UC-11: Сторно

1. Любая проведённая операция → «Сторнировать» → причина обязательна
2. RPC создаёт операцию `reversal`: зеркальные проводки (in ↔ out) **всех** проводок исходной операции, ссылка, исходная помечается `is_reversed`. Частичного сторно не бывает
3. Сторно сторно запрещено; ошибочное сторно исправляется повторным вводом исходной операции с комментарием-ссылкой (правило раздела 8)
4. Частичная отмена уже частично возвращённого платежа — не через сторно, а через дополнительный `refund` (UC-7) на невозвращённый остаток
5. **Полное сторно платежа с уже существующим активным `refund` запрещено на уровне RPC** (инвариант 6): если `net_refunded > 0` (был хотя бы один невозвращённый refund), полное сторно этой операции отклоняется — доступен только дополнительный `refund` на остаток. Если возвратов не было вообще (`net_refunded = 0`), полное сторно разрешено как обычно

#### UC-12: Закрытие ретрита

1. Проверка: нет pending и нет нерешённых disputed-операций по объекту учёта (иначе — решить каждую), долги участников отработаны или осознанно зафиксированы
2. RPC `create_closure` блокирует объект (`fin_accounting_objects`), повторно проверяет пункт 1 под блокировкой (иначе платёж, проведённый в момент проверки, мог бы проскочить мимо snapshot), пересчитывает `totals_snapshot` и создаёт `fin_object_closures(version=1, is_initial=true, status=report_pending, closed_at, closed_by)` — это и есть «финансово закрыт»; денежные операции по объекту уже запрещены, PDF ещё не готов
3. PDF формируется и загружается отдельно из сохранённого snapshot; по готовности `finalize_closure` связывает уже загруженный файл — проставляет `attachment_id`, `finalized_at`, `finalized_by`, `status=finalized`. Сбой генерации не откатывает закрытие и не блокирует ничего денежного — объект остаётся закрытым со статусом «отчёт формируется»
4. После закрытия: обычные операции и правка аналитики по объекту запрещены (под той же блокировкой объекта — включая правку `category_id`/`cost_center_id`/`participant_id`/`participant_balance_kind`, перенос `object_id` и переходы `approval` в/из `disputed`, раздел 5); допускаются `payment`/`refund`/`income`/`expense`/`donation`/`reversal`/`reconciliation_adjustment` с пометкой `is_post_close` и обязательной причиной, а также правка начислений администратором с причиной
5. Любая из этих поздних операций/правок проставляет `fin_accounting_objects.report_dirty_at = now()` — UI показывает «Отчёт требует перевыпуска». Новая версия `fin_object_closures` создаётся не немедленно, а когда администратор нажимает «Перевыпустить PDF»; предыдущие версии сохраняются, не удаляются, `report_dirty_at` обнуляется

#### UC-13: Справочники

CRUD с аудитом: статьи (только админ создаёт), контрагенты, валюты, курсы. Архивирование вместо удаления при наличии привязанных данных.

2. Пометки в ленте: «не проверено» (pending), «оспорено: [причина]» (disputed), «сторнировано: [причина]», «за Кафе» (cost center)

#### UC-17: Guest Portal → «Финансы»

Сторнированные платежи показываются с пометкой «возврат» / «отменён», не исчезают. Пожертвования, внесённые вместе с платежом, показываются отдельной строкой и не входят в расчёт долга.

## 7. Формулы с примерами

```
Остаток счёта = Σ(in) − Σ(out) по всем проведённым проводкам счёта
  Пример: Кафе/₹: выдано 200 000 (in) − траты 57 000 (out) = 143 000

Остаток чекпоинта = Σ(in) − Σ(out) по проводкам счёта с ledger_seq ≤ cutoff_ledger_seq

**Все формулы этого раздела считаются в рамках одной пары `(participant_id, retreat_id)`, никогда по участнику в целом.** Один и тот же участник может одновременно иметь долг по ретриту A и аванс по ретриту B — это не взаимозачитываемые числа, а два независимых расчёта:
```text
fin_charges.retreat_id = выбранный retreat_id
fin_participant_opening_balances.retreat_id = выбранный retreat_id
fin_postings.object_id = fin_accounting_objects.id, где fin_accounting_objects.retreat_id = выбранный retreat_id
```
`general`-кредит участника применяется только к начислениям **того же** ретрита — general-аванс по будущему ретриту B не гасит долг по текущему ретриту A, даже если оба принадлежат одному участнику. Сводная карточка участника (если нужна) показывает отдельные позиции по ретритам, а не единую взаимозачтённую сумму.

Сальдо участника по блокам — алгоритм (порядок фиксированный: org_fee → accommodation → meals → extra), всё в рамках `(participant_id, retreat_id)`:

  1. Для каждого блока k ∈ {org_fee, accommodation, meals, extra} посчитать
     block_debt[k] = charges.net_amount(kind=k, не отменённых)
     − Σ signed amount_base проводок(participant_balance_kind = k)
     + Σ opening(kind=debt, balance_kind=k)
     − Σ opening(kind=credit, balance_kind=k)
  2. Отдельно посчитать общий (не привязанный к блоку) долг и кредит:
     general_debt     = Σ opening(kind=debt, balance_kind=general)
     general_credit   = Σ opening(kind=credit, balance_kind=general)
                       + Σ signed amount_base проводок(participant_balance_kind=general)
     general_remaining = general_credit
  3. general_remaining применяется последовательно по блокам k в порядке
     org_fee → accommodation → meals → extra, затем к general_debt:
       apply = min(general_remaining, max(block_debt[k], 0))
       block_debt[k] −= apply
       general_remaining −= apply
     (после четырёх блоков: apply = min(general_remaining, max(general_debt, 0));
      general_debt −= apply; general_remaining −= apply)
     general не создаёт аванс там, где долга уже нет — шаг просто пропускается
  4. general_remaining после этого (если > 0) = «Общий аванс», к блоку не привязан.
     general_debt после этого (если > 0, ничем не покрыт) = «Общий долг»
  где знак проводки:
    payment.in              → +（уменьшает долг)
    refund.out               → −（восстанавливает долг)
    reversal(payment).out    → −（восстанавливает долг, как refund)
    reversal(refund).in      → +（снова уменьшает долг)
  Пожертвование (participant_balance_kind=none) в расчёт не входит

  Пример 1: начислено 60 000 ₹ (оргвзнос 15 000 + проживание 35 000 + питание 10 000);
  специальный платёж 10 000 ₹ наличными (participant_balance_kind=accommodation);
  general-платёж 500 $ по курсу 83.5 = 41 750 ₹ (participant_balance_kind=general);
  opening-позиций нет (текущий участник, не cutover)

  Шаг 1 (долги по блокам до general):
    org_fee: 15 000 − 0 = 15 000
    accommodation: 35 000 − 10 000 = 25 000
    meals: 10 000 − 0 = 10 000
    extra: 0 − 0 = 0

  Шаг 2: general_debt = 0, general_credit = general_remaining = 41 750

  Шаг 3 (general_remaining применяется по порядку):
    org_fee:        apply = min(41 750, 15 000) = 15 000 → долг 0,   general_remaining = 26 750
    accommodation:  apply = min(26 750, 25 000) = 25 000 → долг 0,   general_remaining = 1 750
    meals:          apply = min(1 750, 10 000)  = 1 750  → долг 8 250, general_remaining = 0
    extra:          apply = min(0, 0) = 0 → долг 0
    general_debt:   apply = min(0, 0) = 0

  Итог: org_fee 0, accommodation 0, meals — долг 8 250 ₹, extra 0, общий долг 0, общий аванс 0.
  (Итоговая сумма долга совпадает с «быстрой» формулой ниже; разбивка по блокам корректна —
  ни один блок не получает искусственный аванс, пока в следующем блоке есть долг)

  Пример 2 (`extra`, чтобы показать, что теперь у него есть собственный долговой блок):
  начисление `kind=extra` (доп. ночь) 5 000 ₹; платёж 5 000 ₹ (participant_balance_kind=general).
  Шаг 1: extra-блок = 5 000 − 0 = 5 000 (остальные блоки без начислений = 0).
  Шаг 3: general_remaining 5 000 применяется по порядку, доходит до extra:
  apply = min(5 000, 5 000) = 5 000 → extra-долг 0, general_remaining = 0.
  Итог: сальдо 0, а не ложный аванс 5 000 — раньше `extra` не имел собственного
  block_debt и general «повисал» без применения.

  Пример 3 (cutover, общий opening-долг без привязки к блоку — не путать с авансом):
  Excel-реестр Ванамали Гопала показывает участника должным 10 000 ₹ без детализации
  по блокам → opening(kind=debt, balance_kind=general, amount=10 000). Начислений
  и платежей после cutover ещё не было. general_debt = 10 000, general_credit = 0.
  general_remaining = 0, применять к блокам нечего (все block_debt = 0). Шаг для
  general_debt: apply = min(0, max(10 000, 0)) = 0 → general_debt остаётся 10 000.
  Итог: «Общий долг» 10 000 ₹ — участнику показывается долг, а не отрицательный
  «аванс» (старый алгоритм не умел обрабатывать kind=debt с balance_kind=general
  и молча терял эту позицию).

Долг участника по конкретному ретриту (быстрая оценка без разбивки по блокам, тот же scope `(participant_id, retreat_id)`, что и выше) =
    Σ активных начислений.net_amount(base) WHERE retreat_id = выбранный
  + Σ opening(kind=debt, base) WHERE retreat_id = выбранный
  − Σ opening(kind=credit, base) WHERE retreat_id = выбранный
  − Σ signed платежей(base) WHERE object_id соответствует выбранному retreat_id, исключая participant_balance_kind=none
  (быстрая формула обязана учитывать fin_participant_opening_balances — иначе после
  cutover она и подробная поблочная формула могут вернуть разные итоги для одного
  участника, что недопустимо: это два представления одного и того же числа)
  Пример: начислено 60 000, заплатил 51 750 (без учёта пожертвования), opening нет → долг 8 250 ₹

Возврат (refund):
  net_refunded = Σ(amount ВСЕХ refund этой проводки) − Σ(amount refund этой проводки, которые сами были сторнированы)
    (одна из двух эквивалентных формул, раздел 5 инвариант 6 — НЕ «активные минус сторнированные»:
    это вычло бы сторнированный refund дважды. Пример: платёж 100, refund 40, затем reversal
    этого refund 40 → верно net_refunded = 40 − 40 = 0, available_to_refund = 100;
    ошибочная формула «активные (=0, т.к. этот refund уже сторнирован) минус сторнированные (40)»
    дала бы −40, а available_to_refund 140 — переплату можно было бы вернуть дважды)
  available_to_refund = original.amount − net_refunded
  refund.amount_base = original.amount_base × (refund.amount / original.amount),
    округление — ROUND(x, 2), для половин от нуля (стандартное поведение PostgreSQL numeric);
    последний возврат, закрывающий available_to_refund полностью, получает точный остаток
    original.amount_base − net_refunded_base_до_этого_refund, а не независимо округлённое
    произведение — иначе сумма частичных возвратов в редких случаях разойдётся на копейку
    с original.amount_base. Где:
      net_refunded_base = Σ(amount_base ВСЕХ refund этой проводки)
                        − Σ(amount_base refund этой проводки, которые сами были сторнированы)
    (та же формула, что и net_refunded в base-выражении — «предыдущие refund» здесь
    НЕ означает «все когда-либо созданные»: сторнированный refund из этой суммы вычитается
    его собственным amount_base, а не входит в неё как есть. Пример: платёж 100 $ = 8 350 ₹;
    refund №1: 40 $ = 3 340 ₹; reversal refund №1: +3 340 ₹ → net_refunded_base = 3 340 − 3 340 = 0;
    refund №2 на весь остаток 100 $ получает 8 350 − 0 = 8 350 ₹, а не 8 350 − 3 340 = 5 010 ₹,
    как дала бы наивная сумма «amount_base всех предыдущих refund-строк без учёта их сторно»)
  refund.currency = original.currency; refund.rate_used наследуется от original (не запрашивается
    заново, не пересчитывается по курсу на дату возврата)

  Пример: платёж 100 $ по курсу 83.5 = 8 350 ₹; возврат 40 $ →
  refund.amount_base = 8 350 × (40/100) = 3 340 ₹ (не пересчитывается по сегодняшнему курсу,
  даже если он стал 84); доступный остаток после возврата = 60 $ / 5 010 ₹;
  повторный возврат суммой > 60 $ от той же проводки → отказ

Подотчёт = остаток custodial-счёта подотчётника
  Пример: Ашиш/₹: перевод 10 000 (in) − расходы по чекам 8 000 (out)
  − возврат 2 000 (out-перевод) = 0

Сальдо ретрита (base) = Σ приходов − Σ расходов по object_id (через fin_accounting_objects);
  переводы между счетами (обе проводки внутри системы) не считаются;
  прасад-пожертвования — приход, в отчёте по статье «Прасад» показываются
  как уменьшение чистого расхода (5 000 расход − 2 000 пожертвований = 3 000 чистыми)

Аналитика подразделения X = Σ signed amount_base проводок, где
  posting.cost_center_id = X И тип операции ∉ {transfer, opening}
  (cost_center_id уже записан в проводку при её создании — раздел 4.1/4.3/4.16, не вычисляется заново
  из текущего account.default_cost_center_id; сам перевод/opening в аналитику подразделений
  не входит, иначе выдача 200 000 департаменту выглядела бы его «доходом»)

  Пример: Кухня купила холодильник 50 000 со своего счёта, посчитанный при вводе cost_center_id = Кафе.
  Остаток счёта Кухня/₹ уменьшился на 50 000 (деньги физически оттуда ушли).
  Аналитика Кухни = её обычные траты (57 000), холодильник в неё НЕ входит.
  Аналитика Кафе = свои траты со своего счёта + 50 000 холодильника = выросла на 50 000.
  Выдача Касса → Кафе 200 000 (`transfer`) ни в чью аналитику не попадает.

Себестоимость на участника = (расходы ретрита по согласованным статьям) / (знаменатель)
  — числитель и знаменатель уточняются с Ванамали Гопалом (открытый вопрос)
```

## 8. Административные правила вместо механизмов

Редкие ситуации решает человек по правилу; след остаётся в аудит-логе. Принцип: если кейс случается реже раза в месяц и его исправление видно в логе — механизм не нужен.

| Ситуация | Правило | След |
|----------|---------|------|
| Отмена платежа, по которому уже был частичный возврат | Дополнительный `refund` на остаток (`original_amount − net_refunded`) той же проводки. Полное `reversal` заблокировано RPC, пока `net_refunded` ненулевой (инвариант 6) | Refund + причина |
| Комиссия PayPal | Две отдельные операции: `payment` на gross + `expense` «Комиссия процессинга» на сумму комиссии, не одна операция с двумя проводками | Payment + Expense |
| Ошибочное сторно | Повторный ввод исходной операции, в комментарии — ссылка на ошибочное сторно | Аудит + комментарий |
| Трата департамента оказалась выдачей под отчёт | Сторно расхода (движения денег не было, это была ошибка классификации) + перевод на custodial-счёт получателя | Reversal + transfer |
| Трата департамента оспорена (`disputed`), деньги ещё физически у ответственного | Перевод с custodial-счёта ответственного обратно на кассу — обычная операция, не сторно | Transfer + причина |
| Трата департамента оспорена (`disputed`), деньги потрачены безвозвратно, ответственный возмещает из личных средств | `income` «Возмещение оспоренного расхода» со ссылкой на disputed-операцию в комментарии (перевод здесь математически неверен — не компенсирует реальную потерю, см. раздел 7) | Income + причина |
| Перенос платежа на другой ретрит: проводка создана **до** initial closure своего объекта | `object_id` можно менять администратору только пока объект открыт и у проводки нет активных refund/reversal (инвариант 2). **После закрытия объекта — никогда**, вне зависимости от каких-либо условий: исправление — только `reversal` (если `net_refunded = 0`) + новый ввод. При наличии активного refund прямая правка запрещена независимо от closure | Аудит-лог правки, либо reversal + новый ввод |
| Перенос платежа на другой ретрит: проводка сама является post-close (создана **после** initial closure своего объекта) | `object_id` можно менять администратору по обычным правилам post-close-аналитики (раздел 5) — блокировка обоих объектов, пересчёт `is_post_close`, `report_dirty_at` новому и/или старому объекту, если они закрыты (см. ниже «Перенос post-close-проводки между объектами») | Аудит-лог правки + `report_dirty_at` |
| Дубли участника в CONTACTS слиты | `contacts.alias_contact_id → canonical_contact_id` в справочнике контактов (вне финмодуля). `participant_id`/`payer_contact_id` на уже проведённых проводках и начислениях **не переписываются** — это нарушило бы неизменяемость и могло бы разорвать связку с существующим refund (инвариант 2). Экраны и поиск разрешают alias в canonical при отображении и фильтрации | Правка в справочнике контактов, не в fin_-таблицах |
| Передача ответственности за custodial-счёт | Одна RPC `transfer_account_responsibility`: проверяет и последний чекпоинт, и текущий `MAX(ledger_seq)` счёта — не только чекпоинт, который новая проводка после сверки не меняет; затем меняет `responsible_person` | Операция + аудит |
| Расхождение сверки не найдено | Один вызов `perform_reconciliation` с причиной: создаёт `reconciliation_adjustment` и чекпоинт атомарно; `original_difference` фиксирует величину расхождения до корректировки (после неё `difference` станет нулём) | Операция + причина |
| При сохранении сверки счёт изменился с момента открытия формы (`reconciliation_stale`) | Форма обновляется, администратор пересчитывает и сохраняет заново | Отказ + повтор |
| Групповой платёж в двух валютах | Два групповых платежа | — |
| Ошибочный курс замечен позже | Сторно затронутых операций + повторный ввод (base фиксируется при проведении — старые отчёты не «плывут», исправляются только явно) | Reversal |
| «Взял из бокса, верну», сумма известна | `donation` на полную пересчитанную сумму + `transfer` касса → custodial-счёт человека на изъятую сумму | Две операции |
| «Взял из бокса, верну», сумма неизвестна | В учёт входит только реально пересчитанное содержимое на момент выемки; неизвестный gross не декларируется | Donation на факт |
| Chargeback / отмена онлайн-платежа банком | Сторно платежа с причиной; реальный счёт может уйти в минус (инвариант 6) | Reversal |
| Поздняя операция по закрытому ретриту (доплата долга, возврат, chargeback, обнаруженная комиссия, поздний документ департамента) | `payment`/`refund`/`income`/`expense`/`donation`/`reversal`/`reconciliation_adjustment` с `is_post_close = true` и обязательной причиной; аналитика уже закрытых проводок не правится | Операция + is_post_close + `report_dirty_at` (новая версия `fin_object_closures` — только по кнопке «Перевыпустить») |
| Начисление создано/отменено/заменено по уже закрытому ретриту | Разрешено только администратору, с причиной (`creation_reason`/`cancelled_reason` + `cancelled_at`/`cancelled_by`) | Аудит-лог + `report_dirty_at` (новая версия `fin_object_closures` — только по кнопке «Перевыпустить») |
| Ошибка в opening-позиции участника (`fin_participant_opening_balances`) обнаружена после даты X | Исходная строка не редактируется и не удаляется. `create_opening_correction` под блокировкой объекта создаёт компенсирующую строку (`corrects_opening_balance_id` → исходная, `kind`/`balance_kind` по смыслу поправки, обязательная `correction_reason`); если ретрит уже закрыт — проставляет `report_dirty_at` | Новая строка + причина + `report_dirty_at` (если закрыт), аудит-лог |
| Ошибка в денежном `opening` счёта (не участника — касса/карта/custodial) обнаружена после запуска | До первого чекпоинта счёта: полный `reversal` ошибочного `opening` + новый replacement `opening`. После первого чекпоинта: только `perform_reconciliation` с `reconciliation_adjustment` — второй `opening` не создаётся никогда | Reversal + opening, либо reconciliation_adjustment |

## 9. Экраны

Раздел **«Финансы»** в главной навигации бэк-офиса (рядом с Housing, Kitchen, CRM, Admin):

```
Финансы
├── Главная
├── ДДС
├── Участники ретрита
├── Счета
├── Сверка
├── Аналитика (по ретриту · общая)
└── Справочники
```

Вне раздела: **«Мои счета»** (пользователь счетов), **Guest Portal → «Финансы»** (участник).

### 9.1 Главная

- Карточки остатков реальных счетов, сгруппированы по group_name («Касса»: ₹ 347 000 · ₽ 12 400 · $ 1 200); метка «сверено по №1842, 15.07» у каждого
- **«Входящие»** — счётчик pending-операций; клик → список на согласование
- **Оспоренные** — счётчик disputed-операций, ждущих решения человека
- **Долги** — участники с задолженностью по текущему/ближайшему ретриту: имя, сумма, ретрит; клик → карточка
- **Минусы** — подотчётные и (после reversal/reconciliation_adjustment) реальные счета в минусе
- Лента последних операций

### 9.2 ДДС

- Журнал операций, разворот в проводки
- Колонки: дата · тип · статья · участник/контрагент · объект учёта · комментарий · приход · расход · остаток (в срезе одного счёта)
- **Цветовая схема по типу операции** (палитра из прототипа Ванамали Гопала); pending и disputed — свои пометки; сторнированные — перечёркнуты с ссылкой; post-close — отдельная иконка
- **Три кнопки:** «↑ Приход» · «↓ Расход» · «⇄ Перевод» — каждая открывает форму с предзаполненным направлением
- Сортировка от новых к старым: в срезе одного счёта — по `ledger_seq` (надёжный порядок, гарантирован блокировкой при вставке). В общей ленте по нескольким счетам — по `operation.created_at DESC, ledger_seq DESC`: `ledger_seq` двух независимых счетов может закоммититься в порядке, отличном от числового (блокировка гарантирует порядок только внутри одного счёта), так что глобальную ленту нельзя сортировать одним ledger_seq — на денежную корректность это не влияет, только на порядок отображения
- Итоговая строка — **параллельно по валютам** без конвертации (₹ / ₽ / $ / € отдельно)
- Синяя строка-разделитель «Итого по ретриту [название]» при группировке по ретритам
- Массовые чекбоксы просмотра — ритуал самоконтроля при сверке
- Фильтры: счёт / группа / ретрит / статья / участник / контрагент / тип / период / статус согласования

### 9.3 Участники ретрита

- Селектор ретрита (включая будущие — для предоплат)
- Таблица: имя · статус оргвзноса · комната · проживание·питание (гостиница + номер, тип питания, даты заезда/выезда, доп. дни) · долг или аванс
- Клик на строку → **раскрываемая карточка, 5 колонок:** оргвзнос / проживание / питание / дополнительно / итог; в каждой: начислено — оплачено — остаток (колонка «дополнительно» скрыта, только если по этому блоку одновременно нет начисления, opening-позиции, проводки с `participant_balance_kind=extra` и ненулевого остатка)
- Чекбоксы нескольких участников → карточки разворачиваются + итог по группе
- Кнопки в строке: **«Платёж»** · **«Изменить»** (начисления) · **«Принять долг»** (форма платежа с предзаполненной суммой долга) · **«Возврат»** (выбор конкретной проводки для возврата)

### 9.4 Счета

- Таб «Реальные»: остатки, чекпоинты (номер и дата)
- Таб «Подотчётные»: департаменты и внешние подотчётники; остаток = подотчёт; счётчик pending и disputed по каждому; клик → лента операций счёта
- Кнопки: «Выдать» (перевод с кассы), «Создать счёт», «Доступ»

### 9.5 Сверка

- Выбор счёта → расчётный остаток на `opened_seq`, зафиксированный при открытии формы
- Наличные: формы пересчёта по точкам хранения и номиналам, автоподсчёт; безнал — поле фактического остатка из выписки
- Разница подсвечена; подсказка последнего чекпоинта (номер проводки + дата)
- **Одна кнопка «Сохранить сверку»** — не две отдельные («чекпоинт» / «корректировка»), чтобы UI-разработчик не завёл два независимых запроса к разным RPC. Поведение зависит только от `difference`, вычисляется на клиенте для UX, но решает всё равно единственный вызов `perform_reconciliation`: `difference = 0` → сохраняет чекпоинт; `difference ≠ 0` → показывает «Расхождение не устранено. Создать корректировку и сохранить чекпоинт?», требует причину, затем тот же вызов `perform_reconciliation` с `adjustment_reason`
- История сверок счёта

## 10. Технические требования

1. Составные операции — Postgres-функции (RPC, SECURITY DEFINER, с фиксированным `search_path`): `create_payment`, `create_transfer`, `create_expense`, `create_income`, `create_refund`, `create_donation`, `create_reversal`, `create_opening`, `create_closure` (блокирует объект, фиксирует `totals_snapshot`, создаёт версию со `status=report_pending`), `finalize_closure` (не «повторяет генерацию» — связывает уже загруженный PDF, проставляет `attachment_id`/`finalized_at`/`finalized_by`/`status=finalized`), `reissue_closure` (post-close перевыпуск: блокирует объект, требует `report_dirty_at IS NOT NULL` — иначе `report_not_dirty` и выход без создания версии; пересчитывает snapshot, создаёт следующую версию, обнуляет `report_dirty_at`). Плюс обязательные, но не денежные: `create_charge`, `cancel_charge`, `set_approval` (переходы `pending`/`approved`/`disputed`, требует object lock при disputed на объектной операции), `update_posting_analytics` (правка `category_id`/`cost_center_id`/`object_id`/`participant_id`/`participant_balance_kind`/`contractor_id` с блокировкой проводки и, при необходимости, обоих объектов; конкретные ограничения по каждому полю в зависимости от наличия активного refund/reversal — инвариант 2), `perform_reconciliation` (единственная публичная точка входа для сверки, раздел 5 п.10), `load_opening_balances` (идемпотентная пакетная загрузка реестра cutover, разово), `create_opening_correction` (единственный способ исправить opening-позицию после cutover — блокирует объект, создаёт компенсирующую строку, при существующей initial closure ставит `report_dirty_at`), `transfer_account_responsibility` (проверяет и последний чекпоинт, и текущий `MAX(ledger_seq)` счёта перед сменой ответственного — раздел 8), CRUD-функции справочников (счета, статьи, курсы, контрагенты, cost centers, доступ). **`create_reconciliation_adjustment` — не отдельная клиентская команда**, а внутренняя SQL-функция, вызываемая исключительно изнутри `perform_reconciliation` в той же транзакции (раздел 5 п.10) — иначе корректировка и чекпоинт могли бы сохраниться раздельными вызовами с промежуточным несогласованным состоянием
2. Прямой DML клиента в fin_-таблицы запрещён; чтение — через views с фильтрацией по fin_account_access
3. Остатки — вычисляемые (view / функция); остаток «по чекпоинту» — отдельная функция с параметром `cutoff_ledger_seq`
4. Клиентские UUID, hash содержимого вычисляется сервером (не принимается от клиента): тот же UUID и тот же вычисленный hash — возврат созданной операции; тот же UUID, другое содержимое — ошибка конфликта (не молчаливый no-op)
5. Аудит-триггер на все fin_-таблицы, кроме `fin_audit_log`; наблюдателю лог отдаётся через маскирующий view
6. Constraint: валюта проводки = валюте счёта; amount > 0; валюта `fin_charges` и `fin_participant_opening_balances` = INR; `fin_exchange_rates` — `UNIQUE NULLS NOT DISTINCT (object_id, effective_date, from_currency)`. Дата события (`occurred_on`) чекпоинтом не ограничена (см. инвариант 5) — ограничение осталось только для `fin_reconciliations.cutoff_ledger_seq`, которое проверяется через блокировку счёта при сохранении (защита от `reconciliation_stale`)
7. `ledger_seq` — bigserial на `fin_postings`, выделяется только после блокировки затронутых счетов (`FOR UPDATE`), индекс `(account_id, ledger_seq)` для быстрого расчёта остатка «на чекпоинт»
8. Блокировка строк счетов (`SELECT ... FOR UPDATE`) в RPC перед проверкой остатка и перед выделением `ledger_seq`, всегда в порядке `ORDER BY account_id` при двух и более счетах — защита от гонки, deadlock и от рассинхронизации seq с порядком коммитов. Блокировка объекта (`fin_accounting_objects`) — тем же принципом, отсортированный порядок при нескольких объектах
9. numeric(14,2), timestamptz, IST для occurred_on. Округление — по сценариям, они не взаимозаменяемы:
   - **Независимые строки мультивалютного платежа** (несколько валют одной операции): каждая строка — `amount_base = ROUND(amount × rate_used, 2)` независимо от других; общий `amount_base` операции = Σ строк. Никакой коррекции остатка между строками разных валют — у них нет единой заранее заданной base-суммы, чтобы под неё что-то подгонять
   - **Перевод между валютами** (`transfer`): `amount_base` входящей ноги = `amount_base` исходящей ноги точно; курс входящей ноги выводится из этого равенства, а не считается независимо
   - **Серия частичных `refund` одной проводки**: последний `refund`, закрывающий `available_to_refund` до нуля, получает точный оставшийся base-остаток `original.amount_base − net_refunded_base` (где `net_refunded_base` считается той же формулой, что `net_refunded`, но в base — Σ amount_base всех refund минус Σ amount_base их reversal, раздел 7), а не собственное округлённое произведение и не сумму «предыдущих refund» без поправки на их сторно
   - **Разделение одной одновалютной суммы по одному курсу на несколько строк** (например, расход 70%/30% на два объекта): первые строки округляются независимо, последняя получает остаток общего `amount_base`, чтобы сумма частей точно совпадала с целым
   Тип округления во всех случаях — PostgreSQL numeric `ROUND` (для половин — от нуля)
10. Экспорт CSV/XLSX из ДДС и аналитики — через те же views (маскирование сохраняется)
11. Backup ежедневно + PITR (Supabase); восстановление — ответственный Нарада. Цели: RPO ≤ 1 час, RTO ≤ 4 часа, тест восстановления раз в квартал. Критерий восстановления: остатки сходятся, `ledger_seq` цел, аудит-лог доступен, последний чекпоинт воспроизводится, **и дополнительно**: у каждой записи `fin_attachments` есть файл в Storage с совпадающим `sha256`; у каждой `finalized`-версии `fin_object_closures` доступен её PDF; orphan-файлов в Storage и orphan-записей `fin_attachments` без файла — нет
12. Полиморфная ссылка `fin_attachments.parent_id` — существование родителя проверяет RPC/триггер, не нативный FK
13. Матрица допустимых проводок по типу операции (для валидации в RPC):

| Тип | Проводки | Ключевые проверки |
|---|---|---|
| `payment` | только `in`, одна и более | payer, participant, object, `participant_balance_kind` — `NOT NULL`, если `participant_id` задан: `org_fee`/`accommodation`/`meals`/`extra`/`general` (гасит долг) или `none` (для строки именного пожертвования, внесённой вместе с платежом — не гасит долг) |
| `refund` | только `out`, ровно одна проводка на одну исходную (для MVP один `refund` = один `refund_of_posting_id`; групповой возврат нескольким участникам = несколько отдельных операций `refund`, не мульти-строчная одна) | `refund_of_posting_id` обязателен, та же валюта что у источника, сумма ≤ `original_amount − net_refunded` под блокировкой источника (инвариант 6). `participant_id`, `participant_balance_kind`, `object_id`, `rate_used` **наследуются сервером** от исходной проводки — клиент их не передаёт и не может переопределить |
| `transfer` | ровно одна `out` + одна `in` | разные счета, `amount_base` обеих равен, `category_id = NULL`, `cost_center_id = NULL`, `participant_id = NULL`, `participant_balance_kind = NULL` |
| `expense` | только `out`, одна и более | `category_id` обязателен; если `participant_id` указан (редкий случай — расход, приписываемый конкретному гостю без изменения его долга), `participant_balance_kind` обязан быть `none` — не может уменьшать долг, это не платёж |
| `income`, `donation` | только `in`, одна и более | `category_id` обязателен; та же защита, что у `expense` — если `participant_id` указан, `participant_balance_kind = none`. Иначе обычный `income` (например, возврат от контрагента) с ошибочно проставленным `participant_id` мог бы случайно погасить чей-то долг |
| `opening` | `in` (реальный/подотчётный счёт) или явно разрешённая `out` | только при инициализации счёта; `participant_id IS NULL`, `participant_balance_kind IS NULL` — участник и его баланс представлены отдельной таблицей `fin_participant_opening_balances` (4.14), а не проводкой |
| `reversal` | полное зеркало всех проводок исходной | одно `reversal` на операцию (частичный уникальный индекс по `original_operation_id`); наследует `category_id`, `cost_center_id`, `participant_balance_kind`, `participant_id`, `object_id`, `contractor_id` источника |
| `reconciliation_adjustment` | одна `in` или `out` | `fin_reconciliations.adjustment_operation_id` обязателен; `participant_id IS NULL`, `participant_balance_kind IS NULL` — расхождение сверки не относится к конкретному участнику |

Стек — существующий стек бэк-офиса: Vanilla JS + DaisyUI + Tailwind + Supabase, без сборки, GitHub Pages.

## 11. Запуск (Cutover)

**Подготовка (до даты X):**
1. Ванамали Гопал составляет реестр: перечень счетов с остатками, деньги на руках департаментов, долги участников, авансы, подотчёты (Ашиш), незакрытые расхождения Excel
2. Незакрытые расхождения — разбор до запуска или явное списание с причиной
3. Заведение справочников: счета, статьи, курсы, cost centers, доступ пользователей
4. Создание `fin_accounting_objects` (type=retreat) для всех активных и ближайших будущих ретритов

**Shadow-период (2 недели до даты X):** двойной ввод — Excel остаётся рабочим, операции дублируются в систему **в той же production-базе**, чтобы Ванамали Гопал и департаменты учились на реальном интерфейсе; ежедневное сравнение остатков. Перед датой X — reset-скрипт по точному списку ниже, затем загружаются opening-позиции по пункту 2 — иначе `opening` лёг бы поверх shadow-операций и задвоил остатки.

**Reset-скрипт перед датой X (точный список, не «все fin_-таблицы»):**
```text
Очищаются (данные shadow-периода, которые нужно тестировать заново):
  fin_operations, fin_postings, fin_reconciliations,
  fin_participant_opening_balances, fin_charges, fin_object_closures,
  вложения и аудит-записи, созданные в shadow-период

Сохраняются (заводились один раз в подготовке, до shadow-периода):
  fin_accounts, fin_categories, fin_currencies, fin_exchange_rates,
  fin_contractors, fin_account_access, fin_accounting_objects, fin_cost_centers
```
Reset не трогает справочники и объекты из подготовки — иначе после очистки не на что было бы грузить opening. Дополнительно: shadow-вложения в Supabase Storage (чеки, тестовые PDF) удаляются вместе с данными; `ledger_seq` (bigserial) перезапускается, чтобы боевая последовательность начиналась с 1, а не продолжала shadow-нумерацию; `fin_accounting_objects.report_dirty_at` сбрасывается в `NULL` (`UPDATE fin_accounting_objects SET report_dirty_at = NULL`) — если в shadow-период тестировали закрытие и post-close операции, поле могло остаться «грязным» уже после того, как все операции и версии закрытий по нему удалены. Это единственное официально допустимое нарушение принципа append-only аудита (раздел 5, инвариант 7) во всём документе — одноразовое, только для миграционного окна cutover, и не применяется ни в каком другом сценарии.

Reset выполняется: в maintenance mode (без активных пользовательских сессий записи), одной контролируемой миграцией, после резервной копии, с проверкой orphan-файлов в Storage по завершении.

**Дата X:**
1. Reset-скрипт (список выше) выполнен, проверка: перечисленные к очистке таблицы пусты, перечисленные к сохранению — на месте
2. `opening`-операции по всем счетам: реальные (Касса/₹ 347 000…) и подотчётные (Кафе/₹ 143 000, Ашиш/₹ 10 000…) — это единственные денежные проводки на дату запуска. Деньги, физически находящиеся на руках у департамента, входят **только** в opening его custodial-счёта, не в opening кассы одновременно — иначе те же деньги посчитаются дважды
3. Долги и авансы участников загружаются по правилу:

```text
Детализированный долг, природа которого известна (оргвзнос/проживание/питание/доп.)
  → fin_charges: kind = org_fee / accommodation / meals / extra

Исторический долг без надёжной детализации по блокам
  → fin_participant_opening_balances: kind = debt, balance_kind = general

Исторический долг с известным блоком, но без данных для полноценного начисления
  (нет unit_price/quantity — просто «должен за проживание N рублей»)
  → fin_participant_opening_balances: kind = debt, balance_kind = конкретный блок

Аванс (переплата) участника
  → fin_participant_opening_balances: kind = credit, balance_kind = блок или general
```

Одна и та же сумма фиксируется только одним из перечисленных способов — никогда двумя. **Авансы участников — записью в `fin_participant_opening_balances` (kind=credit), не денежной операцией `payment`**: аванс уже физически лежит внутри остатка кассы/счёта, отражённого в `opening`-проводке пункта 2; отдельная платёжная запись задвоила бы эти деньги
4. Сверка каждой цифры с Excel — ответственный Ванамали Гопал
5. Боксы не переносятся (содержимое вне учёта до выемки)

**После даты X:** операции — только в системе; Excel — read-only архив. Платежи через CRM дублируются вручную (до интеграции фазы 2).

**Критерий успеха:** три подряд ежедневных чекпоинта без расхождений по всем реальным счетам.

**Rollback:** если система не взлетела — возврат в Excel с выгрузкой не только операций периода, но и новых начислений, opening-позиций, approval-статусов и сверок за период (CSV), разбор причин.

## Приложение: приёмочные сценарии (105)

1. Повторная отправка формы (тот же UUID, тот же вычисленный сервером hash) → одна операция; тот же UUID с другим содержимым → ошибка конфликта
2. Мультивалютный платёж (10 000 ₹ + 500 $) → одна операция, две проводки, долг пересчитан по base
3. Групповой платёж за троих → три проводки, карточки корректны, чужие данные участникам не видны

88. `create_refund` читает `object_id` исходной проводки без блокировки, блокирует найденный объект, затем перечитывает `object_id` уже под блокировкой проводки → если между чтениями кто-то перенёс проводку в другой объект — набор заблокированных объектов неверен, RPC откатывается и повторяет с актуальным набором
89. Вложение с `posting_id`, указывающим на проводку счёта Кафе → видно пользователю с доступом к Кафе; вложение без `posting_id` на операции с проводками по чужому и своему счёту → видно, только если пользователь сам его загрузил или все проводки операции на его счетах
90. Две сверки наличных с одинаковым итогом, но разным порядком ввода номиналов и точек хранения → нормализация `counts` даёт одинаковый `request_hash` — повтор считается идемпотентным, а не новым запросом
91. Счёт с `reconciliation_mode=cash_count` не принимает `statement_balance` в `perform_reconciliation`; счёт с `reconciliation_mode=statement` не принимает `counts` — оба отклоняются как некорректный payload для этого режима
92. `create_opening_correction` с тем же `id`/`request_hash` → возвращает прежнюю компенсирующую строку, не создаёт вторую; тот же `id` с другим payload → `idempotency_conflict`
93. Пользователь счетов пытается создать `expense` по уже закрытому ретриту → отказ `object_closed`; та же поздняя трата, введённая администратором с причиной → успешный post-close `expense`
94. Вложение с `parent_id` = операция A, но `posting_id`, указывающей на проводку операции B → отказ (constraint `fin_postings.operation_id = parent_id`)
95. Post-close проводка создана с ошибочной статьёй → администратор правит `category_id` этой проводки под блокировкой объекта, с причиной → правка проходит и ставит `report_dirty_at`; та же попытка на проводке, проведённой **до** initial closure → отказ (аналитика заморожена навсегда)
96. Сверка счёта зафиксирована на seq=100, затем проведена новая проводка (seq=101) → `transfer_account_responsibility` с `expected_seq=100` отклоняется `account_changed_since_reconciliation`, хотя последний чекпоинт всё ещё формально «100» — потому что проверяется ещё и текущий `MAX(ledger_seq)=101 ≠ 100`
97. Кассовый пересчёт с монетами: `other_amount = 35, other_comment = "мелочь"` в одной из точек хранения → корректно входит в серверно вычисленный `counted_balance`
98. Ошибочный opening обнаружен после cutover → исправляется только `create_opening_correction` (компенсирующей строкой), не создаёт второй `opening`-проводки поверх уже работающего счёта и не редактирует исходную строку
99. В cutover-режиме второй активный `opening` на тот же счёт создан с другим (не тем же) UUID → отказ — идемпотентность по UUID/hash не защищает от двух РАЗНЫХ запросов на один счёт, поэтому проверка «нет активного opening» обязательна независимо от режима
100. Ошибочный денежный `opening` кассы (347 000 вместо 337 000): обнаружен до первого чекпоинта счёта → `reversal` ошибочного `opening` + новый replacement `opening` на верную сумму; обнаружен после первого чекпоинта → только `perform_reconciliation` с `reconciliation_adjustment`, второй `opening` создать нельзя
101. `create_charge` с тем же UUID/`request_hash` после сетевого сбоя (ответ не дошёл до браузера) → возвращает то же начисление 30 000 ₹, не создаёт второе — участник не оказывается должен 60 000 ₹
102. `update_posting_analytics` с устаревшим `expected_audit_version` (проводку кто-то успел поменять между открытием формы и отправкой) → отказ, а не тихая перезапись чужой правки
103. Post-close расход ошибочно отнесён к закрытому ретриту A, администратор переносит его в открытый ретрит B → блокировка обоих объектов, `is_post_close` пересчитывается в `false` (B открыт), `report_dirty_at` проставляется только A (закрытому), не B
104. `update_posting_analytics`, `set_approval` и `transfer_account_responsibility` сохраняют `reason`/`request_id` в `fin_audit_log` через `set_config('app.change_reason', ...)`, а не молча меняют состояние без следа причины
105. `create_opening_correction` с `corrects_opening_balance_id`, ссылающимся на строку другого участника или другого ретрита → отказ (referential-проверка: коррекция обязана быть в рамках того же участника и ретрита, что исходная строка)

*Модель: 16 таблиц, 7 инвариантов. Ключевые принципы: счета от денег, с явным `reconciliation_mode` (cash_count/statement); custodial-модель, включая внешних подотчётников; неизменность денежных полей (включая amount_base/rate_used) + полное сторно (без частичного, заблокировано RPC при активном refund с верной формулой net_refunded, включая её base-версию); аналитика pre-closure проводок заморожена навсегда, post-close проводок — правится администратором под object lock с `report_dirty_at`, включая корректный пересчёт `is_post_close` при переносе между объектами, а после активного reversal заморожена независимо от post-close-статуса; единый порядок блокировок во всех RPC с протоколом «прочитать→заблокировать→перепроверить»; два разных контракта идемпотентности — command-level (UUID/серверный hash) для всех команд создания, включая `create_charge` и `create_attachment`, и state-level (сравнение текущего состояния, включая `expected_audit_version`/проверку по текущему `MAX(ledger_seq)`, а не только по чекпоинту) для команд перехода состояния; сверка — один атомарный RPC `perform_reconciliation(statement_balance | counts)` с серверным пересчётом наличного остатка, поддержкой монет (`other_amount`) и фиксированной нормализацией `counts`; аудит-лог хранит `reason`/`request_id` для state-команд через session-контекст, не только `before`/`after`; согласование с чётким окном изменяемости approval до/после закрытия; снапшот цен в начислениях (сервер вычисляет `amount = ROUND(quantity×unit_price,2)`, не доверяя отдельным полям клиента), только ₹, с четырьмя долговыми блоками (включая `extra`); формулы сальдо участника строго в рамках `(participant_id, retreat_id)`; учётный объект и версионируемое закрытие (`fin_object_closures`) с блокировкой объекта, явным `report_dirty_at`, идемпотентностью и разделением момента закрытия и готовности PDF; денежный `opening` — ровно один активный на счёт без исключений для cutover-режима, исправление — reversal+replacement до первого чекпоинта или reconciliation_adjustment после; `create_opening_correction` с referential-проверками (участник/ретрит/валюта/самоссылка), без клиентского `request_hash`; post-close-операции — только роль Администратор; `participant_balance_kind` ограничен по типу операции; cost center — из справочника `fin_cost_centers`; identity участника и жертвователя — через `payer_contact_id`/`participant_id` с alias→canonical слиянием; вложения — с точной привязкой к проводке (`posting_id`) и referential constraint на согласованность с `parent_id`; передача ответственности за custodial-счёт проверяет и чекпоинт, и текущий `MAX(ledger_seq)`; политика округления и выбора курса детерминированы; политика `occurred_on` для reversal различает ошибку ввода и реальное обратное движение; backup/restore критерии включают целостность вложений и PDF в Storage; полный аудит-лог, не аудитирующий сам себя. Редкие кейсы — административными правилами (раздел 8), не механизмами. Открытые вопросы 1–4 (состав статей, себестоимость, нестандартное размещение, Club 108) не блокируют реализацию, уточняются по ходу с Ванамали Гопалом и АК. Implementation guide с Нарадой.*
